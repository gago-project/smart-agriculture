# 工作台路由化导航与下拉菜单设计

**日期：** 2026-04-22  
**状态：** 已确认  
**范围：** `apps/web`

## 背景

当前工作台头部将 `墒情管理`、`查询日志`、用户名、`退出登录` 以横向按钮/文字直接罗列在右上角，视觉上较拥挤，也不利于后续继续增加工作台入口。

同时，当前页面切换主要依赖 `apps/web/workspace/App.tsx` 里的本地状态 `workspaceView`，并不是基于真实路由。这样会带来几个问题：

- URL 不能表达当前页面位置，刷新和直达体验不清晰
- 权限边界主要停留在前端视图切换层，不够语义化
- 头部菜单和页面内容耦合在一个大组件里，后续扩展成本高

## 目标

- 将头部右侧入口改成一个下拉菜单，替代横向罗列按钮
- 将工作台页面切换改为真实路由，而不是本地 `workspaceView`
- 明确主要路由权限边界
- 保持当前登录方式、现有 API、现有页面主体组件不大改

## 非目标

- 不重构现有登录体系为 cookie session 或 Next middleware 方案
- 不改动墒情管理页、查询日志页、问答页本身的业务能力
- 不新增复杂权限模型，只复用现有 `user / developer / admin` 角色语义

## 方案对比

### 方案一：真实路由 + 共享工作台壳层（推荐）

将页面拆分为真实路由，例如 `/chat`、`/admin`、`/query-logs`，同时把当前 `App.tsx` 从“单体总控”改成“共享工作台壳层 + 插槽内容”的方式。头部统一渲染一个用户下拉菜单，各页面只负责自己的主体内容。

**优点**

- URL 语义清晰，可刷新、可直达、可前进后退
- 权限边界和页面边界一致
- 头部导航可以稳定复用
- 改动范围可控，复用现有 `ChatPanel`、`SoilAdminPage`、`AgentLogPage`

**缺点**

- 需要拆分当前 `App.tsx`
- 需要补若干路由守卫和页面测试

### 方案二：继续单页，只改成 query 参数切换

例如保留一个页面，通过 `?view=chat` / `?view=admin` / `?view=query-logs` 控制内容。

**优点**

- 实现快
- 复用现有结构最多

**缺点**

- 本质仍是“假路由”
- 权限和页面职责仍不够清晰
- 后续扩展菜单项时仍会继续堆逻辑

### 方案三：服务端中间件强守卫

将所有页面权限提升到服务端统一中间件控制。

**优点**

- 权限中心化

**缺点**

- 当前登录态主要依赖前端持久化 token，不适合本次直接切到中间件方案
- 对这次 UI/路由改造来说过重

## 选型结论

采用 **方案一：真实路由 + 共享工作台壳层**。

原因：

- 正好同时解决“导航丑”和“不是路由”的两个问题
- 不需要改动登录基础设施
- 改造后页面职责更清晰，便于继续扩展工作台入口

## 路由设计

### 页面路由

- `/`：工作台入口。若已登录，则跳转到 `/chat`；若未登录，则展示登录页
- `/chat`：问答主页面，所有已登录用户可访问
- `/admin`：墒情管理，仅 `admin` 可访问
- `/query-logs`：查询日志，仅 `admin` 与 `developer` 可访问

### 权限规则

- 未登录用户访问 `/chat`、`/admin`、`/query-logs` 时，显示登录页
- 已登录但权限不足时：
  - 访问 `/admin`：回退到 `/chat`
  - 访问 `/query-logs`：回退到 `/chat`

这里保持当前前端 token 鉴权模式，不引入新会话机制。

## 头部菜单设计

右上角不再横向罗列多个按钮，而是改为一个下拉菜单触发器。

### 菜单内容

- `墒情管理`：仅 `admin` 可见，点击跳转 `/admin`
- `查询日志`：`admin`、`developer` 可见，点击跳转 `/query-logs`
- `用户名：{username}`：仅展示，不触发跳转
- `退出登录`：执行登出，并回到登录态

### 交互要求

- 点击触发器展开 / 收起菜单
- 点击菜单项后自动收起
- 点击当前已在的页面项，不重复切换
- 登出后清理前端登录态
- 菜单应具备基础可访问性语义，例如按钮、菜单容器、菜单项区分

## 组件拆分设计

### 新的共享壳层

新增一个共享工作台壳层组件，职责包括：

- 初始化登录态
- 根据登录状态决定显示登录页还是工作台
- 渲染公共头部
- 渲染公共侧边会话栏
- 承载各路由页面主体内容

建议将当前 `apps/web/workspace/App.tsx` 拆为更偏壳层职责的结构，例如：

- 工作台壳层组件
- 头部用户菜单组件
- 路由页面分别决定主内容

### 页面内容复用

- `/chat`：继续使用 `ChatPanel` + `Composer`
- `/admin`：继续使用 `SoilAdminPage`
- `/query-logs`：继续使用 `AgentLogPage`

这样可以避免改动页面主体实现，只调整承载方式和导航方式。

## 数据流与状态设计

### 鉴权状态

继续复用 `useAuthStore`：

- `initAuth()`：应用初始化时检查 token
- `login()`：登录成功后持有 token 与用户信息
- `logout()`：调用退出接口后清理本地状态

### 路由状态

不再保留 `workspaceView` 本地状态作为页面主切换来源。当前页面由 Next 路由决定。

页面是否可见由：

- 当前 URL
- 当前 `authUser.role`

共同决定。

## 错误处理

- `initAuth()` 失败：按未登录处理，显示登录页
- 用户手工访问无权限页面：自动跳回 `/chat`
- 登出接口失败：仍清空本地登录态，保持当前实现策略
- 在权限收敛过程中避免出现页面闪烁，优先复用当前 `idle/checking/authenticated/anonymous` 状态机

## 测试设计

本次改造应至少补 / 改以下测试方向：

### 文件与路由契约

- 断言 `/chat`、`/admin`、`/query-logs` 页面文件存在
- 更新对旧 `workspaceView` 方案的断言，避免继续依赖状态切页

### 权限与导航契约

- 断言头部菜单根据角色展示不同入口
- 断言开发者可见查询日志但不可见墒情管理
- 断言普通用户不可见查询日志和墒情管理
- 断言登出仍走现有退出接口

### 组件行为

- 断言当前工作台不再横向渲染 `墒情管理 / 查询日志 / 用户名 / 退出登录` 这一组按钮组合
- 断言使用路由跳转而不是 `workspaceView` 本地切换

## 实施原则

- 尽量少改业务组件，优先做壳层和路由层改造
- 权限逻辑集中，避免散落在多个页面里重复判断
- 样式尽量复用现有工作台视觉体系，只新增菜单所需最小样式
- 不顺手改 unrelated 页面和业务逻辑

## 实施后的预期结果

- 工作台头部更简洁，入口集中在用户下拉菜单中
- `/chat`、`/admin`、`/query-logs` 成为清晰的主要工作台路由
- 权限边界更明确：
  - 已登录用户：问答
  - `developer` / `admin`：查询日志
  - `admin`：墒情管理
- 现有业务组件基本保持不动，改动主要集中在壳层、路由和头部导航
