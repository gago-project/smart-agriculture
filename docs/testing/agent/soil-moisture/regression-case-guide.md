# 墒情 Agent 回归样例维护指南

本指南定义 `soil-moisture` Agent 的回归测试应如何分层、扩展和沉淀，并明确正式 Case 只维护一套主库。

## 1. 回归目标

- 保障 `input_type -> intent -> route -> answer_type` 主链路稳定。
- 保障 `ExecutionGate`、地区解析、规则判断、模板输出、查询日志等关键控制点不回退。
- 让新增问题类型、新增模板、新增别名解析规则都能快速补进统一测试体系。

## 2. 三层测试资产

### 2.1 冒烟层

目标：快速判断服务是否还能正常处理核心查询。

建议至少覆盖：

- 1 个概览问题
- 1 个地区详情问题
- 1 个设备异常问题
- 1 个模板输出问题
- 1 个兜底/澄清问题

适用时机：

- 本地改动后快速自测
- 部署前后快速验证
- 线上问题修复后的最小复测

### 2.2 验收层

目标：验证当前 MVP 公开承诺的能力边界。

当前基准入口：

- 正式 Case 主库：`testdata/agent/soil-moisture/case-library.md`
- 验收说明：`docs/testing/agent/soil-moisture/acceptance-test-matrix.md`

适用时机：

- 里程碑验收
- 功能合并到主分支前
- 对商务、产品展示前

### 2.3 专项回归层

目标：针对高风险链路单独加密测试，避免“主流程能跑、边角回退”。

建议专项：

- 地区别名与轻度模糊解析
- 多轮上下文继承与衰减
- `ExecutionGate` 超限拦截
- 模板严格模式与解释模式
- 查询日志写入 / 不写入边界
- 无数据 / 找不到对象 / 公共异常兜底

## 3. 什么情况必须补 Case

出现以下任一情况，都应补回归样例：

- 新增了 `input_type`、`intent` 或 `answer_type`
- 新增了 SQL 模板、查询路由或门禁规则
- 调整了地区解析、简称补全、模糊匹配逻辑
- 调整了预警模板、异常判断规则或展示口径
- 修复了线上 Bug，且 Bug 可被稳定复现
- 调整了“是否写查询日志”的策略

## 4. Case 设计要求

每个 Case 至少要能回答以下问题：

- 用户到底怎么问？
- 是否依赖上下文？
- 预期命中哪个 `intent`？
- 是否允许查库？
- 预期走哪个 `query_type / SQL 模板`？
- 回答应归属哪个 `answer_type`？
- 哪个关键断言最值得检查？

扩展 Case 时不追求各分类均分，应按业务价值、真实问法频率和回归风险加权。墒情概览、排名对比、地区/设备详情、异常分析、预警模板输出优先加密；非业务、安全提示、能力边界只保留能覆盖关键行为的代表性样例。

## 5. 文档与数据的边界

- `docs/testing/agent/soil-moisture/`：写测试规则、验收口径、评审说明。
- `testdata/agent/soil-moisture/`：未来存放可长期维护的 Case 源数据。
- `outputs/`：只存当次执行生成的 Excel、CSV、截图，不承载长期规则。

## 6. 推荐维护方式

- 先更新 `docs/testing/agent/soil-moisture/acceptance-test-matrix.md` 中的验收规则或说明口径。
- 再把正式 Case 的新增、删减、修订直接落到 `testdata/agent/soil-moisture/case-library.md`。
- 每次线上问题修复后，至少补 1 条对应专项回归样例。
- 每次对商务批量发测前，先跑一轮冒烟层和验收层。
