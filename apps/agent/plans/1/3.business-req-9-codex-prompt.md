## 背景与业务规则

用户问法示例：
- "最近30天全省预警处置情况怎么样？"
- "上周南通市的预警处置进度如何？"
- "4月份徐州市处理了多少条预警？"

回答模板（格式固定，不得修改）：
```
{时间范围}{区域范围} 内共出现 {X} 条墒情预警信息，处置情况如下：
已处理 {X1} 条，待处理 {X2} 条，超时已处理 {X3} 条，超时待处理 {X4} 条。
```

无数据时：
```
{时间范围}{区域范围} 内未查询到有效墒情预警信息，无对应处置数据。
```

**强制规则**：四种处置状态的名称和展示顺序永远固定为：
`已处理 → 待处理 → 超时已处理 → 超时待处理`
绝对不能调整顺序或修改名称。

`pub_status` 字段语义（已业务侧确认）：

| pub_status | 含义 | 对外展示 |
|---|---|---|
| 1 | 待处理 | 待处理 |
| 2 | 超时未处理 | 超时待处理 |
| 3 | 已处理 | 已处理 |
| 4 | 超时处理 | 超时已处理 |

Agent 直接读 `pub_status` 字段，**不重新计算超时**。

```sql
-- sys_region: 行政区划表，精确到乡镇（9位编码）

  id           VARCHAR(36)  NOT NULL,

  created_at   DATETIME         NULL,
  updated_at   DATETIME         NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_region_code (region_code),
  INDEX idx_parent_code (parent_code),
  INDEX idx_region_level (region_level)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='行政区划表';
```

### 步骤 2：导入脚本

**新建文件**：`scripts/import/import_sys_region.py`

完整脚本内容参见 plan §3.1。核心要点：
- 需要 `xlrd==1.2.0`（xlrd 2.x 不支持 `.xls` 格式）
- 通过 `argparse` 接收 `--file` 参数
- `region_level` 推断规则：xls 有 level 字段直接用；否则按编码位数推断（2位→1省，6位→2市/县，9位→4乡镇）
- 用 `FIELD_MAP` 方式灵活映射 xls 表头名称
- `ON DUPLICATE KEY UPDATE` 保证幂等执行
- DB 连接使用环境变量 `MYSQL_HOST/PORT/USER/PASSWORD/DATABASE`

**新建文件**：`scripts/import/import_warning_disposal.py`

完整脚本内容参见 plan §3.2。核心要点：
- 使用 `openpyxl`（`.xlsx` 格式）
- 只导入 `warn_type == "墒情预警"` 的行（虫害预警等一律排除）
- city/county 解析优先级：
  1. 解析 `address` 字段（格式"苏州市昆山市"→city=苏州市，county=昆山市）
  2. 解析失败时通过 `region_code` 查 `sys_region` 表
- `--truncate` 选项支持幂等重跑
- `ON DUPLICATE KEY UPDATE` 仅更新 `pub_time`、`pub_status`、`do_advice`

### 步骤 3：Repository 层修改

**修改文件**：`apps/agent/app/repositories/soil_repository.py`

在现有 `query_warning_records_async` 方法（当前约 line 898）之后，新增以下三个方法：

**[3-1] 静态方法 `build_warning_disposal_audit_sql`**
- 参数：`city, county, start_time, end_time`（均为 `Optional[str]`）
- 返回：审计日志用的 SQL 字符串（literal SQL，非参数化）
- SQL 结构：
  ```sql
  SELECT COUNT(*) AS total,
         SUM(pub_status=3) AS status_done,
         SUM(pub_status=1) AS status_pending,
         SUM(pub_status=4) AS status_overtime_done,
         SUM(pub_status=2) AS status_overtime_pending
  FROM warning_disposal_record
  WHERE pub_status IN (1,2,3,4) [AND city=... ...]
  ```
- 注意：字符串值要做单引号转义防止 SQL 注入

**[3-2] 同步方法 `query_warning_disposal_stats`**
- 参数：`city, county, start_time, end_time`（keyword-only，`Optional[str]`）
- 返回 dict（键名固定）：
  ```python
  {"total": int, "已处理": int, "待处理": int, "超时已处理": int, "超时待处理": int}
  ```
- DB 字段映射：
  - `pub_status=3` → `已处理`
  - `pub_status=1` → `待处理`
  - `pub_status=4` → `超时已处理`
  - `pub_status=2` → `超时待处理`
- 表不存在（MySQL 1146 / "doesn't exist"）时返回 `total=0` 的空统计，不抛异常
- 其他 DB 错误抛 `DatabaseQueryError`
- 使用 `_connect()` 模式（与现有代码一致）

**[3-3] 异步包装 `query_warning_disposal_stats_async`**
- 用 `asyncio.to_thread()` 包装 `query_warning_disposal_stats`
- 传递相同的 keyword 参数

```python
@staticmethod
def _is_warning_disposal_query(text: str) -> bool:
    """
    预警处置查询：用户关心的是预警是否被处理/处置进度，而非预警事件本身。
    必须含处置主题词，或含预警词 + 处置意图词。
    规则相关问题优先返回 False（由 warning_rule 路由处理）。
    """
    has_disposal_subject = any(t in text for t in _WARNING_DISPOSAL_SUBJECT_TOKENS)
    has_warning_word     = any(t in text for t in _WARNING_RECORD_QUERY_TOKENS)
    has_disposal_intent  = any(t in text for t in _WARNING_DISPOSAL_INTENT_TOKENS)
    has_rule_word        = any(t in text for t in _WARNING_RULE_QUERY_TOKENS)
    if has_rule_word:
        return False
    return has_disposal_subject or (has_warning_word and has_disposal_intent)
```
