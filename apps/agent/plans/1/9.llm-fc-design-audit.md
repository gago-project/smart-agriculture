# 9. LLM + Function Calling 设计审计与整改建议

> 审计日期：2026-04-27  
> 审计范围：当前 `apps/agent/app/` 主链路实现与 `plans/1/*.md` 正式设计文档  
> 说明：本审计**不讨论当前环境是否已搭建完成**，只评估设计质量、代码与文档一致性、数据查询智能体能力边界与风险点。

---

## 1. 结论先行

### 1.1 设计评分（忽略环境搭建）

| 维度 | 评分 | 结论 |
|------|------:|------|
| 架构方向 | 8/10 | 已经从“规则驱动问答引擎”走向“受限数据查询 Agent” |
| 数据边界设计 | 7/10 | 有明确 `facts only` 理念，但工程约束还未闭环 |
| Function Calling 设计 | 6/10 | 方向正确，但工具语义与执行语义尚未收紧 |
| 多轮上下文设计 | 6/10 | 引入了真实消息历史，但上下文标准化与边界控制不足 |
| 文档与实现一致性 | 5/10 | 主骨架一致，核心业务契约与若干细节明显漂移 |
| 综合评分 | 7/10 | 方向正确、实现未收口，适合继续迭代而不是回退 |

### 1.2 一句话判断

当前实现已经具备了“LLM 决策 + Function Calling + 受限 Flow”的正确骨架，但还没有真正达到“生产级数据查询智能体”的收口状态。

**最需要立刻补强的红线：**

> **域内业务问题不能允许模型在未命中工具的情况下直接输出最终答案。**

这是当前实现中风险最高的一点，因为系统目标明确要求回答必须建立在真实数据之上，不能凭空生成。

## 3. 代码与文档对齐情况

## 3.1 已对齐的部分

以下部分代码与文档基本一致：

| 项目 | 文档口径 | 当前实现 | 结论 |
|------|----------|----------|------|
| 5 节点主链路 | 固定 5 节点 Flow | `routes.py` / `orchestrator.py` 已对齐 | 一致 |
| LLM 决策循环 | `AgentLoopService` 驱动 Function Calling | `agent_loop_service.py` 已实现 | 一致 |
| 时间锚点 | 以最新业务时间为锚点 | `AgentLoopNode` 查询 `latest_business_time_async()` 后注入 prompt | 一致 |
| 工具调用上限 | 最多 5 次 tool iteration | `MAX_TOOL_ITERATIONS = 5` | 一致 |
| 参数安全边界 | `top_n`、时间跨度、工具白名单限制 | `ToolExecutorService` 已实现 | 一致 |
| Redis 历史 | 使用 `session_msg:{session_id}` | `SessionContextRepository` 已实现 | 大体一致 |

## 3.2 明显不对齐的部分

以下是当前最需要注意的“文档写了，但 live path 没做到”的地方。

### A. `intent` / `answer_type` 仍被文档视为当前契约，但成功路径并未落地

文档中仍把 `intent`、`answer_type` 描述为当前有效契约，甚至写成由 `AgentLoopNode` 推断设置；但当前成功路径中，`AgentLoopNode` 只写：

- `answer_bundle.final_answer`
- `query_result.records`

只有 fallback 时才会写 `answer_type = fallback_answer`。

**结果：**

- 域内成功回答可能出现 `intent=None`
- 域内成功回答可能出现 `answer_type=None`
- 前端、日志、评测、证据链无法基于稳定的结构化语义工作

### B. `DataFactCheck` 文档职责远强于当前实现

文档写的是：

- 核对数字、时间、地区、设备号
- 防止空回答、错对象、内部术语泄漏

当前实现只做了两件事：

- 回答不能为空
- 严格模板模式下，回答里要出现 SN

这与“事实核查节点”的设计目标明显不匹配。

### C. 返回模型暴露了大量旧契约字段，但新主链路并未填充

`ChatResponse` 仍暴露：

- `merged_slots`
- `context_used`
- `business_time`
- `execution_gate_result`
- `query_plan`
- `rule_result`
- `template_result`
- `advice_result`

但当前 `SoilAgentService.achat()` 主路径只稳定返回了：

- `input_type`
- `intent`
- `answer_type`
- `final_answer`
- `query_result`
- `node_trace`
- `final_status`

而且其中 `intent` / `answer_type` 还可能为空。

这意味着：

- 对外 API 契约没有收口
- 前端看到的字段比后端实际产出的多
- 旧版行为测试大量失败并不是偶然，而是结构性漂移

### D. `agent_query_log` 仍在设计文档中重要存在，但新主链路未真正接通

当前服务层会尝试写 `final_state.query_log_entries`，但主链路并没有生成这些 entries。

这会导致：

- 查询日志审计能力弱化
- 复盘和回归定位困难
- 无法形成完整的“用户问题 -> 工具调用 -> 查询结果 -> 最终回答”证据链

### E. `block_end` 已写入设计文档，但 live path 没有任何节点会走到它

当前终态文档仍包含 `block_end`，但路由表、节点声明和 orchestrator 都没有使用该终态。

这是明确的遗留项。

## 3.3 明显遗留的旧架构代码

以下旧架构组件仍保留在仓库中，但**不再接入当前主链路**：

- `IntentSlotService`
- `ContextService`
- `ExecutionGateService`
- `SoilQueryService`
- `TemplateService`
- `SoilRuleEngineService`
- `AdviceService`
- `ConversationBoundaryService`
- `TimeService`

当前不建议继续以“兼容保留”的方式长期共存。

### 建议结论

对于这些**已不在 live path 中、且新主链路完成后可被替代**的旧组件，应明确按“迁移完成后删除”处理，而不是长期挂在仓库里作为所谓兼容层。

### 风险后果

如果继续保留，会带来至少三个问题：

1. 读代码的人很容易误判“系统实际上还在使用旧架构”
2. 文档写的是新设计，测试和返回模型却还在沿用旧契约字段，导致认知混乱
3. Claude / Codex / Cursor 等协作 Agent 会继续把这些旧组件当成现行实现进行分析、修改和补测试

### 删除原则

- 新链路等价能力补齐后，旧服务直接删除，不做长期双轨保留
- 若某个旧组件短期内仍被少量测试或脚本引用，应在同一轮迁移中同步改掉引用，再删除源文件
- 不建议用 `legacy_`、`deprecated_`、`compat_` 的方式长期挂着不删

## 4. 当前 7 个 Tool 的真实语义审计

## 4.1 名义上的 7 个 Tool

当前定义如下：

- `get_soil_overview`
- `get_soil_ranking`
- `get_soil_detail`
- `get_soil_anomaly`
- `get_warning_data`
- `get_advice_context`
- `diagnose_empty_result`

## 4.2 实际执行语义

按当前 `ToolExecutorService` 的实现，这 7 个 tool 并没有真正落成 7 类执行能力。

| Tool | 当前执行语义 | 是否真正独立 |
|------|--------------|--------------|
| `get_soil_overview` | 按条件拉原始 records | 否 |
| `get_soil_ranking` | 按条件拉原始 records，再附带 `top_n` / `aggregation` | 半独立 |
| `get_soil_detail` | 按条件拉原始 records | 否 |
| `get_soil_anomaly` | 按条件拉原始 records，仅多一个时间跨度限制 | 否 |
| `get_warning_data` | 按条件拉原始 records | 否 |
| `get_advice_context` | 按条件拉原始 records | 否 |
| `diagnose_empty_result` | 做存在性诊断 / 时间窗诊断 | 是 |

## 4.3 结论

从当前实现看，真实上只有 3 类执行模式：

1. **查原始 records**
2. **查 ranking 元数据 + 原始 records**
3. **空结果诊断**

也就是说，当前 7 个 tool 更像是“按回答话术拆分的业务标签”，而不是“按真实执行能力拆分的工具”。

这会导致两个问题：

- 模型看到的 tool 边界比真实系统更细
- Function Calling 的判断成本上升，但选错 tool 的后果又不够清晰

## 4.4 更合理的收缩建议

如果按当前产品语义收缩，建议收缩成 4 类：

- `query_soil_summary`：墒情概览查询
- `query_soil_ranking`：墒情排名查询
- `query_soil_detail`：墒情详情查询
- `diagnose_empty_result`：空结果诊断

其中：

- `warning` 更适合作为**回答模式 / 模板模式**
- `advice` 更适合作为**回答模式 / 解释模式**
- `anomaly` 更适合作为 `summary` / `detail` 的一种数据视角，除非后端确实有独立异常检测逻辑

## 5. 关键风险清单

## 5.1 P0：域内问题可以不调用工具直接回答

这是当前设计里最严重的风险。

### 风险描述

`AgentLoopService` 只要收到模型返回的 `text`，就会把它直接当成最终答案接受。  
当前系统中**没有硬约束要求“域内业务问题至少命中一次 tool 才能结束”**。

### 为什么这是首要红线

你的系统目标非常明确：

- 事实必须来自真实数据
- 不能编造
- 不能凭空给出结论

如果允许域内问题在 0 次工具调用的情况下直接结束，那么下面这些都会成为可能：

- 模型凭经验写出“最近整体偏干”
- 模型用聊天历史猜出地区、设备、时间窗
- 模型在没有查数据的前提下输出“建议浇水”或“未发现异常”

### 结论

这条必须作为**第一优先级整改项**。

---

## 5.2 P0：工具语义与执行语义不一致

### 风险描述

Tool 名称看起来是：

- 概览
- 排名
- 详情
- 异常
- 预警
- 建议

但执行层里大多数最后都是同一种 records 查询。

### 风险后果

- 模型难以学会稳定的 tool 选择策略
- 调错 tool 也未必立刻暴露，反而更难发现
- 文档和测试会误以为系统有更细的能力边界

---

## 5.3 P0：返回的是大量原始 records，而不是面向问题的结构化结果

### 风险描述

当前 `summary`、`ranking` 等名义上的高层能力，返回的仍是大批原始 records。

### 风险后果

- token 成本高
- LLM 需要自行做统计/筛选/排名
- 数值漂移、遗漏、排序错误风险上升
- 随着窗口增大，回答稳定性显著下降

### 本质问题

后端把本该确定性完成的“聚合 / TopN / 统计 / 诊断”工作推给了模型。

---

## 5.4 P1：当前消息历史不是标准 `assistant/tool` 转录

### 风险描述

文档说 Redis 保存 OpenAI messages 格式完整历史，但当前实际保存的是：

- `user`
- `assistant`

工具调用和结果只是塞进 assistant 记录里的附加字段，并没有真正保存标准 `tool` message。

### 风险后果

- 下一轮上下文不够标准
- 难以精确复现 agent 执行轨迹
- 后续如果换模型或接入更严格的 function-calling 协议，兼容性差

---

## 5.5 P1：多轮上下文目前过度依赖模型天然理解，没有显式边界

### 风险描述

旧版有显式的上下文继承、衰减、换题边界控制；新版主路径把这部分几乎全部交给了消息历史 + 模型理解。

### 风险后果

- “那里”“那个设备”“上周那个情况”可能串题
- 新问题和追问的边界不清
- 无法稳定控制“继承多少轮、继承哪些槽位”

### 注意

这不是说必须回到旧版槽位继承，而是说明：

> 当前从“显式控制”直接切到“完全依赖模型”，中间缺了一层轻量边界约束。

---

## 5.6 P1：地区别名与时间解析在新主链路中缺少确定性标准化

### 风险描述

当前 live path 中：

- 地区别名 / 轻度模糊解析没有稳定接入
- 时间窗解析主要依赖 prompt 指导模型自己计算

### 风险后果

- 同一个地区多种问法的稳定性下降
- “最近 7 天”“上周”“昨天”等窗口可能被不同模型版本解释得不一致
- Alias、模糊拼写、多候选澄清能力变弱

---

## 5.7 P1：只支持单个 tool call，无法处理一次响应中的多个 tool call

### 风险描述

`QwenClient.call_with_tools()` 只读取 `tool_calls[0]`。

### 风险后果

- 如果模型一次规划多个工具，只有第一个会被执行
- 设计图里“多个独立问题可以依次调用多个工具”与实现并不完全一致

---

## 5.8 P1：空结果诊断只是 prompt 建议，不是强制执行规则

### 风险描述

system prompt 要求“首次空结果后调用 `diagnose_empty_result`”，但代码没有硬约束。

### 风险后果

- 模型可能直接输出模糊解释
- 无法保证“没数据”场景一定能区分为：对象不存在 / 时间窗无数据 / 仅当前条件无数据

---

## 5.9 P1：Query Log 与证据链未闭环

### 风险描述

当前审计日志写入接口还在，但 live path 没有实际生成 `query_log_entries`。

### 风险后果

- 无法稳定复盘错误回答
- 无法做 case 对照和精细化回归
- 无法给业务方展示“这条回答究竟查了什么”

---

## 5.10 P2：`warning` / `advice` / `anomaly` 的分层不清晰

### 风险描述

当前系统把它们同时当作：

- tool 维度
- answer_type 维度
- 业务能力维度

### 风险后果

- 分层职责容易混乱
- 后端 executor 和前端回答模式耦合过深

### 更合理的分层

- tool 负责取什么数据 / 做什么确定性计算
- answer mode 负责怎么组织答案
- template / advice 负责输出风格与解释方式

---

## 5.11 P1：Case Library、QA Skill 与测试入口仍锚定旧架构口径

### 风险描述

当前测试资产和 QA 指导材料仍然大量继承旧架构语义，包括但不限于：

- `testdata/agent/soil-moisture/case-library.md` 仍按旧的一级能力分类维护
- 大量 Case 元数据仍以 `intent` / `answer_type` / `ExecutionGate` / `query_type / SQL` 为核心断言
- `soil-moisture-qa` skill（`.claude/`、`.codex/`、`.agents/` 三份）仍按旧分类、旧回答类型、旧门禁规则组织回归
- `.cursor/rules/soil-moisture-qa.mdc` 仍把旧 QA 口径当作默认规则入口
- `plans/1/README.md` 与 `testdata/agent/soil-moisture/README.md` 仍指向本地不存在的 `docs/testing/agent/soil-moisture/`

### 风险后果

- 实现已经开始向“4 类真实 Tool + 输出模式”迁移，但测试资产仍把系统往旧架构拉回去
- 后续让 Claude / Codex / 其他协作 Agent 改代码时，会被旧 skill 持续灌输过时约束
- 即使实现正确，Case 和回归报告也可能因为旧契约字段失真而产生大量假失败
- 新架构落地后，测试资产无法准确证明“业务回答是否基于真实数据”

### 结论

这不是“把 130 个 Case 简单删成 4 类”的问题，而是：

- 保留高价值真实问法
- 重写 Case 元数据和断言口径
- 把旧的“异常 / 预警 / 建议”从一级查询能力，迁移为输出模式或结果视角

## 6. 修改建议（按优先级）

## 6.1 P0：强制“域内问题至少命中一次工具”

这是当前第一优先级。

### 建议

- 在 `AgentLoopService` 或 `DataFactCheck` 增加硬规则：
  - 若 `InputGuard` 已判定为业务问题
  - 且最终回答前 `tool_calls_made == 0`
  - 则不得直接进入成功终态
  - 必须转入 fallback 或强制要求模型调用工具

### 目标

确保所有域内事实回答都基于真实数据，而不是模型自由生成。

---

## 6.2 P0：把 Tool 收紧成真实语义不同的 4 类

### 建议目标 Tool

- `query_soil_summary`
- `query_soil_ranking`
- `query_soil_detail`
- `diagnose_empty_result`

### 设计原则

- 一个 tool = 一种真实执行能力
- 不按“回答长什么样”拆 tool
- 按“后端到底做了什么确定性工作”拆 tool

---

## 6.3 P0：让 `summary` / `ranking` / `detail` 返回问题导向的结构化结果

### 建议

- `summary` 返回统计结果，而不是 raw records
- `ranking` 返回已聚合、已排序、已裁剪的 TopN 列表
- `detail` 返回单对象详情与必要的最新记录/趋势摘要
- `diagnose_empty_result` 返回标准化诊断枚举与解释字段

### 目标

把确定性工作留在后端，把语言组织留给 LLM。

---

## 6.4 P1：明确 `answer_type` / `intent` / `query_log_entries` 的当前定位

### 两种可选路线

#### 路线 A：继续保留

那就必须在主链路里真正补齐：

- `intent`
- `answer_type`
- `query_log_entries`
- `query_result provenance`

#### 路线 B：正式废弃

那就要同步清理：

- 返回模型
- 前端消费逻辑
- 文档中的旧语义承诺
- 历史测试中的旧契约断言

### 当前建议

优先选 **路线 A**，因为这套字段对审计、前端展示、回归测试仍然有价值。

---

## 6.5 P1：将历史消息标准化为真正的 `assistant/tool` transcript

### 建议

- Assistant 产生 tool call 时保存标准 `assistant` message
- Tool 返回结果时保存标准 `tool` message
- 不再只把 `tool_calls` / `tool_results` 作为 assistant 附属字段保存

### 目标

让多轮上下文更接近标准 FC transcript，减少上下文歧义。

## 6.6 P1：补一层轻量上下文边界，而不是完全依赖模型

### 建议

不必回退到旧版重槽位体系，但至少应补一层轻量约束，例如：

- 是否允许无实体追问直接继承
- 最近几轮内是否允许继承时间窗
- 新完整问题是否重置旧 frame
- 纯指代问题在无清晰历史时是否强制澄清

### 目标

在“完全规则化”和“完全依赖模型”之间找到更稳的中间层。

---

## 6.7 P1：把 alias / time normalization 拉回确定性层

### 建议

- 地区 alias、唯一高置信模糊命中、多候选澄清仍应由代码做
- 时间窗落地最好仍有一层确定性校验或解析结果标准化

### 目标

减少模型版本变化带来的不稳定性。

## 6.8 P1：接通 Query Log 证据链

### 建议

在新主链路中补齐：

- query plan 摘要
- tool 调用参数
- 执行结果摘要
- 最终 answer_type / intent
- request_text / response_text

### 目标

让“用户问题 -> 工具 -> 数据 -> 回答”形成可追溯闭环。

## 6.9 P1：迁移 Case Library、QA Skill 与测试入口

### 建议

把 QA 资产迁移视为本次架构升级的一部分，不要等实现改完后再补。

#### Case Library 迁移原则

- 不要简单把 `130` 个 Case 粗暴删成 `4` 类
- 保留真实用户问法、典型上下文和高价值坏 Case
- 重写元数据与断言，使其从“旧 intent / answer_type / SQL 模板口径”迁移到“新 Tool / 输出模式 / 数据真实性口径”

#### 推荐的 Case 组织视角

可以按 3 层组织，而不是只按 Tool 名称平铺：

- 会话层：非业务 / 澄清 / 边界 / 结束语 / 多轮上下文
- 查询层：`query_soil_summary` / `query_soil_ranking` / `query_soil_detail` / `diagnose_empty_result`
- 输出层：普通数据回答 / anomaly focus / warning mode / advice mode / fallback

#### 旧分类建议映射

- 旧 `概览` -> 新 `query_soil_summary`
- 旧 `排名对比` -> 新 `query_soil_ranking`
- 旧 `地区 / 设备详情` -> 新 `query_soil_detail`
- 旧 `异常分析` -> `summary/detail + anomaly focus`
- 旧 `预警模板输出` -> `detail/summary + warning mode`
- 旧 `指标解释 / 指导建议` -> `detail/summary + advice mode`
- 旧 `无数据 / 找不到 / 兜底` -> `diagnose_empty_result`

#### 建议新增的关键字段

- `是否域内业务问题`
- `是否必须命中 tool`
- `预期 tool`
- `预期输出模式`
- `是否必须查真实数据`
- `真实性断言`
- `结构化证据断言`

#### 必须同步迁移的 QA 入口

- `.claude/skills/soil-moisture-qa/SKILL.md`
- `.codex/skills/soil-moisture-qa/SKILL.md`
- `.agents/skills/soil-moisture-qa/SKILL.md`
- `.cursor/rules/soil-moisture-qa.mdc`
- `plans/1/README.md`
- `testdata/agent/soil-moisture/README.md`

### 目标

让实现、测试、协作提示词、验收口径保持同一套新架构语义，避免“代码按新设计走，Case 和 Skill 还在教旧逻辑”。

## 6.10 P1：旧方案文档、旧兼容代码、旧测试契约应明确删除，而不是长期保留

### 建议

把“删除旧架构遗留物”视为本次升级的正式工作内容，不要只改 live path。

#### 应明确处理的对象

- 不再接入主链路的旧服务与旧节点依赖
- 围绕旧 `intent / answer_type / ExecutionGate / SQL 模板` 的兼容测试
- 仍在传播旧实现心智模型的 README、历史方案稿、skill、rule
- 本地已不存在却仍被引用的测试目录与入口说明

#### 处理原则

- **能删则删**：只要新链路已经覆盖能力，就直接删除旧文件和旧测试
- **不要堆“兼容层”**：不建议继续保留大量“仅供参考”的旧实现
- **删除时成组处理**：代码、测试、文档、skill、rule 一起删或一起迁移，避免留下悬挂引用
- **历史方案稿若保留，只能明确标注为 archived**，不能继续作为执行依据

#### 建议优先删除 / 清理的范围

- 已不在主链路中的旧服务族
- 与旧服务绑定的旧单测 / 契约测试
- 指向旧测试目录或旧方案路径的文档索引
- 仍要求旧 `intent / answer_type / ExecutionGate` 口径的 QA 规则

### 目标

避免仓库在视觉上和语义上长期维持“两套 Agent 并存”的假象。

## 6.12 P2：实施前建议一次性明确的 6 个口径

如果想尽量减少后续反复补充说明，建议在开工前一次性明确以下 6 件事。

### 1. `intent / answer_type / query_type` 是否继续作为正式对外契约

建议默认：

- `intent` 保留
- `answer_type` 保留
- `query_type` 保留，但改成与新 Tool / 新查询语义一致

这样前端、日志、评测、真实性测试都更容易收口。

### 2. 一级 `answer_type` 是否继续保留 11 类

建议默认：

- 不再把当前 `11` 个业务回答模式继续当作最终的一级 `answer_type`
- 一级 `answer_type` 收口为 `5` 类：
  - `soil_summary_answer`
  - `soil_ranking_answer`
  - `soil_detail_answer`
  - `guidance_answer`
  - `fallback_answer`
- 额外拆出 3 层更稳定的辅助契约：
  - `output_mode`：`normal` / `anomaly_focus` / `warning_mode` / `advice_mode`
  - `guidance_reason`：`clarification` / `safe_hint` / `boundary` / `closing`
  - `fallback_reason`：`no_data` / `entity_not_found` / `tool_missing` / `tool_blocked` / `fact_check_failed` / `unknown`

#### 旧 11 类到新结构的建议映射

- `soil_summary_answer` -> `soil_summary_answer + output_mode=normal`
- `soil_ranking_answer` -> `soil_ranking_answer + output_mode=normal`
- `soil_detail_answer` -> `soil_detail_answer + output_mode=normal`
- `soil_anomaly_answer` -> `soil_summary_answer / soil_ranking_answer / soil_detail_answer` 之一 + `output_mode=anomaly_focus`
- `soil_warning_answer` -> `soil_summary_answer / soil_detail_answer` 之一 + `output_mode=warning_mode`
- `soil_advice_answer` -> `soil_summary_answer / soil_detail_answer` 之一 + `output_mode=advice_mode`
- `clarification_answer` -> `guidance_answer + guidance_reason=clarification`
- `safe_hint_answer` -> `guidance_answer + guidance_reason=safe_hint`
- `boundary_answer` -> `guidance_answer + guidance_reason=boundary`
- `closing_answer` -> `guidance_answer + guidance_reason=closing + conversation_closed=true`
- `fallback_answer` -> `fallback_answer + fallback_reason=*`

#### 必须同步修改的文档与测试资产

- `plans/1/2.answer-types-business.md`
- `plans/1/1.plan.md`
- `plans/1/4.python-flow-design.md`
- `plans/1/8.flow-risk-contract.md`
- `plans/1/README.md`
- `apps/agent/README.md`
- `testdata/agent/soil-moisture/case-library.md`
- `testdata/agent/soil-moisture/README.md`
- `.claude/.codex/.agents/.cursor` 下的 soil-moisture QA skill / rule

这样可以把“回答类型”和“输出模式 / 引导原因 / 兜底原因”拆开，避免后续继续把 `warning / advice / anomaly / clarification / boundary / closing` 这些概念误用成一层级的执行契约。

### 3. `warning / advice / anomaly` 的最终定位

建议默认：

- 不是第一层 Tool 分类
- 作为输出模式 / 结果视角保留
- 允许前端继续展示，但不能再要求后端为它们各自维持独立底层查询工具

### 4. Case Library 是否保留现有 CaseID

建议默认：

- 保留现有 `CaseID`
- 不整体推倒重编编号
- 只迁移分类、元数据和断言口径

这样最利于历史回归和商务沟通 continuity。

### 5. 真正用于“答案真实性校验”的结构化证据字段是什么

建议默认至少保留或新增：

- `query_result`
- `query_log_entries`
- `tool_calls_made` / tool trace
- `answer_facts` 或等价的结构化事实摘要

否则真实性测试会被迫依赖脆弱的自然语言比对。

### 6. 本地不存在的 `docs/testing/agent/soil-moisture/` 是要恢复，还是彻底移除引用

建议默认：

- 不再恢复一个新的空 testing 目录来兼容旧引用
- 直接改掉所有悬挂引用
- 以 `testdata/agent/soil-moisture/case-library.md` + QA skill 作为新的正式入口

### 结论

如果以上 6 点都按推荐默认执行，那么后续通常不需要再额外补充高层口径，直接进入实现与验收即可。

## 7. 推荐整改顺序

建议按以下顺序推进：

1. **先补 P0 红线**：域内问题必须命中工具
2. **收缩或重构 Tool 语义**
3. **让后端返回结构化 summary / ranking / detail 结果**
4. **补齐 `answer_type` / `intent` / `query_log`，并把一级 `answer_type` 收口到 5 类**
5. **标准化消息历史 transcript**
6. **补轻量上下文边界**
7. **补 alias / time normalization**
8. **迁移 Case Library、QA Skill 与测试入口**
9. **明确删除旧方案文档、旧兼容代码与旧测试契约**
10. **最后统一收口 README、系统总览、历史计划稿的残留引用**

## 8. 最终判断

当前版本不是“做错了方向”，而是：

> **已经换上了正确的 Agent 骨架，但还没把工具语义、事实红线、上下文边界和审计闭环真正收紧。**

如果只选一件事先做，建议就是：

> **先把“域内问题必须命中工具”做成硬约束。**

因为这是整个数据查询智能体可信度的根基。
