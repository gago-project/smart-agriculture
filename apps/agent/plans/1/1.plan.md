# 1. 墒情 Agent 当前实施方案

## 目标

当前墒情 Agent 只做一件事：围绕真实墒情数据给出受控、可核验、可追溯的回答。

- 事实来源只认 `fact_soil_moisture`
- 查询时间统一使用 `create_time`
- 原始 Excel 字段名是唯一数据契约
- 多轮继承坚持规则优先
- LLM 只做受控补强，不参与事实编造

## 当前数据契约

### 事实表原则

- 墒情事实唯一表：`infra/mysql/init/001_init_tables.sql` 中的 `fact_soil_moisture`
- 原始字段直接保留：`id`、`sn`、`gatewayid`、`sensorid`、`unitid`、`city`、`county`、`time`、`create_time`、各层水分温度与字段状态、`lat`、`lon`
- 管理追溯字段只保留：`source_file`、`source_sheet`、`source_row`
- 查询执行、排序、最新业务时间、时间过滤全部使用 `create_time`
- `time` 只作为原始字段保留，不作为当前 Agent 查询时间列

### 配套表

- `soil_import_job` / `soil_import_job_diff`：导入预览与差异快照
- `region_alias`：`city` / `county` 两级别名映射
- `metric_rule`：规则配置
- `warning_template`：模板配置
- `agent_query_log`：查询审计

## 固定主流程

```text
用户问题
  -> InputGuard
  -> IntentSlotExtract
  -> ConversationBoundary
  -> HistoryContextMerge
  -> TimeResolve
  -> RegionResolve
  -> ExecutionGate
  -> SoilDataQuery
  -> SoilRuleEngine
  -> TemplateRender / AdviceCompose
  -> ResponseGenerate
  -> DataFactCheck
  -> AnswerVerify
  -> Final Answer
```

## 节点职责

### InputGuard

- 识别问候、乱输、越界、纯结束语
- 非业务输入不进查库链路
- 纯结束语命中时清空上下文

### IntentSlotExtract

- 解析 `intent`、`answer_type`
- 抽取当前支持槽位
- 优先规则解析，必要时才调用 LLM 结构化兜底
- 对 LLM 返回值做白名单清洗

### ConversationBoundary

- 判断当前轮是继承、转换、重置还是澄清
- 输出 `inheritance_mode`、`boundary_reason`
- 决定是否沿用上一轮对象、查询框架和时间窗

### HistoryContextMerge

- 只做合并，不做重新判意图
- 当前轮显式槽位覆盖历史同名槽位

### TimeResolve

- 把自然语言时间解析成 `start_time` / `end_time`
- 相对时间以库内最新业务时间为锚点
- 沿用上下文时保持上一轮已解析绝对时间窗，避免漂移

### RegionResolve

- 只解析 `city` / `county` / `sn`
- 地区别名优先走 `region_alias`
- 重名或多候选时进入澄清

### ExecutionGate

- 控制时间跨度、范围规模、TopN 上限
- 只根据 `start_time` / `end_time` 计算跨度
- 阻断后不得继续查询

### SoilDataQuery

- 只执行固定查询模板
- `filters` 只允许 `city`、`county`、`sn`
- 所有 SQL 排序和时间过滤都基于 `create_time`

### SoilRuleEngine

- 基于查询结果临时计算业务状态
- 允许输出运行时分值用于排序解释
- 不把运行时派生字段回写数据库

### TemplateRender / AdviceCompose

- 模板回答走受控模板渲染
- 建议回答只建立在真实事实与规则结果之上

### ResponseGenerate

- 优先确定性回答
- 可选使用 LLM 做受控润色

### DataFactCheck / AnswerVerify

- 核对回答中的数字、时间、地区、设备号
- 防止空回答、错对象、半成品模板和内部实现术语外泄

## 当前时间契约

### 输入语义

- `今天` / `昨天` / `前天`
- `N天前`（N 天前的完整一天）
- `N天前的前M天`（以 N 天前为锚点，向前 M 天的完整窗口）
- `最近N天` / `近N天` / `过去N天`
- `过去N周` / `近N周`（折算为 N×7 天）
- `最近一个月`（滚动 30 天）
- `上周`（上一个完整自然周 Mon–Sun）
- `本周`（本周一至最新业务时间）
- `上个月`（上一个完整自然月）
- `本月`（本月一日至最新业务时间）
- `N月份` / `N月`（指定月；年份自动推断：月份 ≤ 当前月取今年，否则取去年）
- `YYYY-MM-DD`
- `YYYY-MM-DD之前N天`（N 天窗口，以指定日期为结束边界，含当天）
- `YYYY-MM-DD之后N天`（N 天窗口，以指定日期为起始边界，含当天）
- `今年以来`
- `过去2/3/5年`

### 解析结果

统一产出：

- `resolved_time_range`
- `start_time`
- `end_time`
- `time_basis`
- `resolution_mode`

锚点型时间窗（`anchor_before_N_days` / `anchor_after_N_days`）不依赖库内最新业务时间，
`time_basis` 输出为 `anchor_date`，`resolution_mode` 输出为 `anchor_window`。

相对锚窗口（`n_days_ago_N` / `relative_before_M_at_N_ago`）和日历周期标签均依赖最新业务时间，
`resolution_mode` 输出为 `relative_window`。

`calendar_month` 标签需额外的 `target_month` slot（字符串 `"1"`–`"12"`），年份在 resolve 阶段自动推断。

### 规则

- 没有显式时间且没有可继承时间时，才补默认时间窗
- 多轮继承时优先保留上一轮绝对时间窗
- 所有范围最终都落到 `start_time` / `end_time`

## 查询模板

- `SQL-01`：概览
- `SQL-02`：排名
- `SQL-03`：地区 / 设备详情
- `SQL-04`：异常候选
- `SQL-05`：预警模板
- `SQL-06`：建议 / 指标解释
- `SQL-07`：空结果排查

## 多轮上下文契约

Redis 中保存最近 5 轮业务上下文，核心结构为：

- `entity_context`：`city`、`county`、`sn`
- `query_frame`：`query_family`、`intent`、`metric`、`top_n`、`render_mode`、`audience`
- `resolved_window`：`start_time`、`end_time`、`time_label`、`time_explicit`
- `base_query_family`
