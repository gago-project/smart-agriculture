# 5. 墒情 Agent Python 实施计划（受限 Flow 版）

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 基于自研受限 Flow、FastAPI、Redis、MySQL 和强约束业务规则，落地一个可运行、可调试、可测试的墒情 Agent Python 服务。

**Architecture:** 采用单 Agent + 显式 Flow 主链路编排，Flow 节点只负责 orchestration，业务逻辑下沉到 Service 层，数据库访问下沉到 Repository 层。Redis 作为最近 5 轮业务上下文与短期运行态的权威来源，不承担完整运行图恢复语义。LLM 只用于结构化意图提取与受控回答生成。系统按“API → Flow → Service → Repository → DB/Redis/LLM”分层落地，并为后续 Advice Flow、预警增强、专家知识增强预留边界。

**Tech Stack:** FastAPI、Pydantic v2、pydantic-settings、SQLAlchemy 2.x AsyncEngine、asyncmy、Redis、Jinja2、structlog、pytest、httpx、DashScope / 千问 SDK

---

## 0. 前置约束

### 0.1 目标代码仓结构

本计划假定将要创建的 Python 项目根目录为：

`soil-moisture-agent/`

### 0.2 开发原则

- 所有 DB 查询统一走 Repository 封装方法
- 所有 Flow 节点不得直接写 SQL
- 当前实现不引入 `LangChain` 与 `LangGraph`
- Redis 保存最近 5 轮业务上下文，但不保存整份 `FlowState`
- 多轮上下文继承以 `SessionContextStore` 读出的白名单上下文为准
- `ExecutionGate` 必须在查库前执行
- `AnswerVerifyNode` 必须有失败短路与兜底能力
- 当前阶段先落单 Agent 主 Flow，不做多 Agent

### 0.3 开发顺序

## 14. 上下文继承与失败降级

### Task 14: 补齐多轮与硬降级

**要求：**

- Redis 不可用时退回单轮模式
- 模型不可用时退回固定意图词典 + 固定话术
- DB 查询失败时安全报错，不编造事实
- 规则失败时只给事实，不给风险判断
- 模板失败时退回普通结构化回答
- 同一 `session_id` 写上下文时必须带 `turn_id`
- 旧轮次不得覆盖新轮次
- 知识增强不得参与计数、TopN、趋势统计、预警等级判断
