# 8. 墒情 Agent Flow 风险契约与补强方案

> 本文档用于补充 `1 / 4 / 5 / 6 / 7` 中受限 Flow 方案的工程风险约束。  
> 目标不是推翻现有架构，而是把“Flow 少了框架兜底后，需要我们自己补的安全契约”写清楚。

---

## 1. 文档目标

本文件主要回答 4 个问题：

- 受限 Flow 相比 LangGraph，新增了哪些工程风险
- 这些风险最容易以什么 bug 形式出现
- 当前设计必须补哪些硬约束
- 实现阶段最少要补哪些测试

一句话结论：

> Flow 方向是对的，但必须补齐“路由安全、状态安全、重试安全、上下文安全、知识边界安全”这 5 类契约，否则实现时很容易出现稳定性 bug。

---

## 2. 风险总览

### 2.1 当前风险评级

| 风险项 | 当前等级 | 说明 |
|---|---|---|
| 路由漏配风险 | 高 | 自研 Flow 不会像图框架那样天然帮你检查全部边 |
| 状态污染风险 | 高 | `state_patch` 合并若无约束，容易覆盖嵌套对象 |
| 重试死循环风险 | 中高 | `DataFactCheck -> ResponseGenerate` 天然存在回路 |
| 多轮串题风险 | 中高 | Session 上下文若继承条件不严，容易误继承 |
| 知识污染事实风险 | 高 | 专家知识增强后，解释内容可能污染事实回答 |
| 并发上下文覆盖风险 | 中 | 同一 `session_id` 并发请求会相互覆盖 |
| 降级出口不一致风险 | 中高 | 不同节点各自兜底，用户会看到不一致结果 |

---

## 3. 必须补的 5 类契约

## 3.1 路由安全契约

### 风险描述

自研 Flow 中，节点返回 `next_action` 后，是否有合法去向，全靠我们自己保证。

### 典型 bug

- 节点返回了 `retry_response`，但路由表没有登记
- 节点拼错 `next_action`
- `fallback_end` 被某些节点直接终止，绕过统一兜底节点

### 硬约束

- 每个节点必须声明允许返回的 `next_action` 枚举
- `routes.py` 必须覆盖所有合法 `next_action`
- Flow 启动时必须执行：
  - 节点声明与路由表一致性校验
  - 路由目标合法性校验
  - 终止动作集合合法性校验
- `fallback` 不得直接等于终止态，必须走 `FallbackGuard`

### 推荐规则

```text
业务节点
  -> fallback
FallbackGuard
  -> fallback_end
```

---

## 3.2 状态安全契约

### 风险描述

`FlowState` 很强，但如果 `state_patch` 是随意 dict merge，最终会把状态写坏。

### 典型 bug

- `answer_bundle` 被整个覆盖，导致之前字段丢失
- `query_result` 被错误节点改写
- `rule_result` 被 Response 层覆盖

### 硬约束

- 每个节点必须有“状态写入白名单”
- 嵌套对象不得做无约束整对象覆盖
- `merge_state_patch()` 必须按字段级规则合并
- 关键事实字段只允许这些节点写：
  - `query_result`：`SoilDataQueryNode`
  - `rule_result`：`SoilRuleEngineNode`
  - `template_result`：`TemplateRenderNode`
  - `advice_result`：`AdviceComposeNode`

### 推荐实现

- 为每个节点定义 `allowed_patch_fields`
- `merge_state_patch()` 中发现越权字段时直接报错

---

## 3.3 重试与循环安全契约

### 风险描述

Flow 中天然存在回路：

- `DataFactCheck -> ResponseGenerate`

如果没有硬上限，就会无限重试。

### 典型 bug

- 回答草稿总是轻微不一致，系统反复重写
- 某节点异常后一直 fallback / retry 循环

### 硬约束

- `FlowState` 必须包含：
  - `step_count`
  - `retry_count`
  - `node_visit_counts`
- `FlowRunner` 必须设置：
  - `max_steps`
  - `max_retry_count`
  - `max_node_visits_per_node`
- 任一阈值超限时，直接进入 `FallbackGuard`

### 推荐默认值

```text
max_steps = 32
max_retry_count = 2
max_node_visits_per_node = 4
```

---

## 3.4 上下文安全契约

### 风险描述

当前设计已经把上下文白名单收紧了，但还要进一步避免“继承过度”和“并发覆盖”。

### 典型 bug

- 上轮是南京，这轮只问“最近趋势”，系统默认继续用南京，但用户其实换了话题
- 同一 `session_id` 的两次请求并发，旧轮次把新轮次上下文覆盖

### 硬约束

- `SessionContextStore` 只保存：
  - `domain`
  - `region`
  - `time_window`
  - `entity_reference`
  - `last_intent`
- 继承必须经过 `ContextService` 判定，不允许直接拿 Redis 值就写回 `merged_slots`
- 同一 `session_id` 写入必须带 `turn_id`
- 只允许“新 turn 覆盖旧 turn”，不允许旧 turn 回写覆盖新 turn

### 继承策略建议

- 强继承：
  - 明确追问词，如“那上周呢”“这个设备呢”
- 弱继承：
  - 只补地区或时间窗
- 必须澄清：
  - 同时影响对象 + 时间 + 输出方式时

---

## 3.5 知识边界安全契约

### 风险描述

专家知识库是未来高价值能力，但也是最容易污染事实层的地方。

### 典型 bug

- 用户问“近 30 天有几次预警”
- 系统把知识库里的常见情况混进计数回答
- 或者把“通常建议”写成“当前事实”

### 硬约束

- 知识库只参与：
  - 原因解释
  - 处置建议
  - 背景补充
- 知识库不得参与：
  - 精确计数
  - TopN 排名
  - 趋势统计
  - 预警等级判断
  - 结构化事实题

### 推荐实现

- `AdviceComposeNode` 前必须有 `knowledge_eligibility_check`
- eligibility 不通过时，知识层完全不参与

## 4. 必须修正的设计点

### 4.1 `FallbackGuard` 不能被绕过

当前实现应统一改成：

```text
DataFactCheck failed
  -> fallback
  -> FallbackGuard
  -> fallback_end
```

而不是直接：

```text
DataFactCheck failed
  -> fallback_end
```

### 4.2 路由表必须补全线性节点

以下线性节点不能依赖“默认继续”这种隐式逻辑：

- `intent_slot_extract`
- `history_context_merge`
- `time_resolve`
- `region_resolve`
- `soil_data_query`
- `advice_compose`
- `response_generate`

### 4.3 `FlowRunner` 启动时必须做自检

至少校验：

- 节点注册完整性
- 路由完整性
- `next_action` 合法性
- 终止动作合法性

### 4.4 `merge_state_patch()` 必须强类型

不能直接用：

```python
state.__dict__.update(state_patch)
```

必须：

- 字段级 merge
- 越权字段校验
- 嵌套对象 merge 规则校验

---

## 5. 必须新增的测试

### 5.1 Flow 框架层测试

- route table 漏配时启动失败
- 非法 `next_action` 时启动失败
- 未注册节点时启动失败
- `max_steps` 超限时进入 `FallbackGuard`
- `retry_count` 超限时进入 `FallbackGuard`

### 5.2 状态安全测试

- 节点越权写字段时失败
- 嵌套对象错误覆盖时失败
- `query_result` 不允许被非查询节点改写

### 5.3 多轮与上下文测试

- 强继承样例通过
- 弱继承样例通过
- 高风险继承场景转澄清
- 并发写上下文时旧 turn 不覆盖新 turn

### 5.4 知识边界测试

- 事实题中知识层不参与
- 建议题中知识层可参与
- 预警等级判断不允许读取知识库结果

---

## 6. 结论

- 当前架构不需要推倒重来
- `Flow` 方向仍然是更适合这个项目的
- 但必须补齐：
  - 路由安全
  - 状态安全
  - 重试安全
  - 上下文安全
  - 知识边界安全

一句话结论：

> 这次不是重构架构，而是把受限 Flow 从“能跑”补成“能稳”。
