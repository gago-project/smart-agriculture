# 4. 墒情 Agent Python 工程方案

## 设计结论

当前工程坚持：

- 单 Agent
- LLM + Function Calling
- 5 节点受限 Flow
- 原始字段直通
- `create_time` 统一时间契约
- LLM 是决策者，代码是守门员
- 管理员查询证据以 `agent_query_log` 审计链路为准，Web 右侧侧栏走接口按轮次回拉

## 分层结构

```text
API
  -> Orchestrator (5 节点)
  -> Nodes (InputGuard, AgentLoop, DataFactCheck, AnswerVerify, FallbackGuard)
  -> Services (AgentLoopService, ToolExecutorService, InputGuardService, FactCheckService, AnswerVerifyService)
  -> Repositories (SoilRepository, SessionContextRepository)
  -> LLM Adapter (QwenClient + SOIL_TOOLS + SystemPrompt)
```

## 目录角色

- `apps/agent/app/api/`：HTTP 接口
- `apps/agent/app/flow/`：节点、路由表、编排器
- `apps/agent/app/services/`：业务逻辑、工具执行、Agent 循环
- `apps/agent/app/repositories/`：MySQL / Redis 访问
- `apps/agent/app/schemas/`：`FlowState`、枚举、DTO
- `apps/agent/app/llm/`：QwenClient、工具 Schema、System Prompt

## deterministic `/chat-v2` 顶层查询路由

- `DataAnswerService.reply()` 不再自己拼 `summary / latest_record / detail / list / group / count / compare / field` 的顶层布尔判断
- 顶层查询入口统一收口到 `TurnRouteDecisionService`
- 该服务先做轻量归一化，再产出内部 `QueryShape`：
  - `subject`: `soil / rule / template / unsupported_derived`
  - `action`: `summary / detail / list / group / count / compare / field / guidance`
  - `grain`: `record / device / region / city / county / entity / none`
  - `mode`: `standalone / contextual / action_target / explicit_detail / safe_hint`
- `QueryProfileResolverService` 在 `QueryShape` 之后补齐唯一执行画像 `QueryProfile`，当前至少统一以下维度：
  - `data_focus`
  - `answer_mode`
  - `result_grain`
  - `measure / projection / compare_mode`
  - `time_window / slots / follow_up_mode`
- `DataAnswerService` 只负责：
  - 取 `entities / time_evidence / action_result`
  - 调用中心路由分类与 `QueryProfile` 解析
  - 按 `QueryProfile` 分发执行器与渲染
- 真实问法变体与轻量错字不直接扩正式 56 Case，优先补到中心路由矩阵单测
- 当前 `WarningPredicateService` 已作为共享预警判定入口接入 deterministic 查询执行层；`warning_only` 不再只是一层文案标签，而是可被 `summary / list / group / count / compare` 复用的数据焦点
- `QueryProfile` 的正式治理说明见 [`9.query-profile-governance.md`](./9.query-profile-governance.md)

## Orchestrator

编排器只负责三件事：

- 按固定顺序驱动节点
- 根据节点动作跳转
- 在终态前统一收口日志与异常

## AgentLoopService 内部循环

```text
load_history(session_id)
inherited_time_window = latest(history.start_time/end_time)
time_evidence = TimeWindowService(user_input, latest_business_time)
build messages = [system_prompt, ...history, user_message]
tool_hit = False
for i in range(MAX_TOOL_ITERATIONS=5):
    response = qwen.call_with_tools(messages, SOIL_TOOLS)
    if response.type == "text":
        if is_in_domain_business_question and not tool_hit:
            return AgentLoopResult(final_answer=FALLBACK_TOOL_REQUIRED, is_fallback=True)
        save_message_turn(session_id, turn_id, ...)
        return AgentLoopResult(final_answer=response.content)
    if response.type == "tool_call":
        try:
            resolved_args = ParameterResolverService(
                raw_args=tool_args,
                time_evidence=time_evidence,
                inherited_time_window=inherited_time_window,
            )
            if resolved_args.should_clarify:
                return AgentLoopResult(final_answer=clarify_message, needs_clarify=True)
            result = tool_executor.execute(tool_name, resolved_args)
            tool_hit = True
            persist resolved_args to history tool_calls.arguments
            append tool result + query_log_entry to messages
            query_log_entry must include:
              - deterministic query_id = session_id:turn_id:tool_index
              - query_type / query_plan_json / filters_json / time_range_json
              - executed_sql_text (repository audit SQL)
              - executed_result_json (real tool result payload)
        except ToolValidationError as e:
            if e is time-related:
                return clarification_answer
            append error to messages (LLM can adjust and retry)
return AgentLoopResult(final_answer=FALLBACK_MAX_ITER, is_fallback=True)
```

## FlowState 核心字段

| 字段 | 来源节点 | 说明 |
|------|----------|------|
| `request_id` / `session_id` / `turn_id` | 请求注入 | 标识 |
| `user_input` | 请求注入 | 原始用户输入 |
| `input_type` | InputGuardNode | 输入分类 |
| `answer_bundle.final_answer` | AgentLoopNode | 最终回答 |
| `query_result` | AgentLoopNode | 结构化事实载体（summary / ranking / detail / diagnosis） |
| `answer_type` | AgentLoopNode / DataFactCheckNode / FallbackGuard | 一级回答类型（固定为 5 类） |
| `output_mode` | AgentLoopNode | 正常 / anomaly / warning / advice 输出视角 |
| `guidance_reason` | InputGuardNode / AgentLoopNode | clarification / safe_hint / boundary / closing |
| `fallback_reason` | AgentLoopNode / DataFactCheckNode / FallbackGuard | no_data / entity_not_found / tool_missing 等 |
| `conversation_closed` | InputGuardNode | 是否关闭话题 |
| `node_trace` | FlowRunner | 执行路径 |

## 回答契约

一级 `answer_type` 统一为：

- `soil_summary_answer`
- `soil_ranking_answer`
- `soil_detail_answer`
- `guidance_answer`
- `fallback_answer`

辅助字段：

- `output_mode`
- `guidance_reason`
- `fallback_reason`

这样可以把“查询类型”和“输出风格 / 引导原因 / 兜底原因”拆开，避免继续堆积一层级的回答枚举。

## SessionContextRepository 接口

```python
async def load_history(session_id: str) -> list[dict]
async def save_message_turn(
    session_id: str, turn_id: int, *,
    user_message: str,
    assistant_message: str,
    tool_calls: list[dict],
    tool_results: list[dict],
) -> None
async def clear_context(session_id: str) -> None
```

Redis key: `session_msg:{session_id}`，最大 20 条消息，TTL 7200 秒。

## Repository 边界

### SoilRepository

- 只读 `fact_soil_moisture`
- 只暴露 `city`、`county`、`sn` 过滤
- 时间过滤统一是 `create_time`
- SQL 排序统一是 `ORDER BY create_time DESC`

### SessionContextRepository

- 目标是保存标准 `assistant/tool` transcript 或与之等价的结构化历史
- 清空话题时删除 Redis key

### QueryLogRepository

- 只记录真正执行过的查询
- 记录 `raw_args_json`、`resolved_args_json`、`time_range_json`、`entity_confidence`、`time_confidence`、`resolver_warnings`
- 每条 tool 调用写入确定性 `query_id=session_id:turn_id:tool_index`
- `executed_sql_text` 是真实审计 SQL；`executed_result_json` 是该次工具真实返回结果
- 管理员工作台通过 `GET /api/admin/agent/query-evidence?session_id=&turn_id=` 按轮次读取，而不是依赖本地缓存保存完整结果

## 管理员查询证据侧栏契约

- 只对 `admin` 角色开放
- Web 聊天页右侧显示“查询概览 + SQL + 结果数据”
- 前端本地只持久化：
  - 会话消息
  - 当前选中的 assistant 消息 ID
  - 轻量 `session_id / turn_id / should_query`
- 完整证据不写入 localStorage；刷新页面后按 `session_id + turn_id` 重新回拉
- 老日志缺少 `executed_sql_text` 或 `executed_result_json` 时，前端必须提示“历史日志不完整”，并尽量回退到当前消息里的结构化 `query_result`
