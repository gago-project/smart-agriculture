# 4. 墒情 Agent Python 工程方案

## 设计结论

当前工程坚持：

- 单 Agent
- 受限 Flow
- 规则优先
- 原始字段直通
- `create_time` 统一时间契约
- LLM 仅在受控节点参与

## 分层结构

```text
API
  -> Orchestrator
  -> Nodes
  -> Services
  -> Repositories
  -> Schemas / Contracts
  -> Optional LLM Adapter
```

## 目录角色

- `apps/agent/app/api/`：HTTP 接口
- `apps/agent/app/flow/`：节点与路由表
- `apps/agent/app/services/`：业务规则、上下文、时间、响应装配
- `apps/agent/app/repositories/`：MySQL / Redis 访问
- `apps/agent/app/schemas/`：`FlowState`、枚举、DTO
- `apps/agent/app/templates/`：Jinja 模板

## Orchestrator

编排器只负责三件事：

- 按固定顺序驱动节点
- 根据节点动作跳转
- 在终态前统一收口日志与异常

不负责：

- 直接查库
- 直接算规则
- 直接改写回答

## FlowState 核心字段

- 请求基础：`request_id`、`trace_id`、`session_id`、`turn_id`、`user_input`
- 识别结果：`input_type`、`intent`、`answer_type`
- 槽位：`raw_slots`、`merged_slots`
- 时间：`business_time`
- 上下文：`recent_context`、`context_used`、`context_to_save`
- 查询：`query_plan`、`query_result`、`query_log_entries`
- 规则与回答：`rule_result`、`template_result`、`advice_result`、`answer_bundle`
- 收尾：`final_status`、`node_trace`

## 节点边界

### 规则节点

- `InputGuardNode`
- `ConversationBoundaryNode`
- `HistoryContextMergeNode`
- `TimeResolveNode`
- `RegionResolveNode`
- `ExecutionGateNode`

这些节点坚持规则式实现，不接入 LLM。

### 可选 LLM 节点

- `IntentSlotExtractNode`
- `ResponseGenerateNode`

规则解析失败或需要受控润色时才调用。

## Repository 边界

### SoilRepository

- 只读 `fact_soil_moisture`
- 只暴露 `city`、`county`、`sn` 过滤
- 时间过滤统一是 `create_time`
- SQL 排序统一是 `ORDER BY create_time DESC`

### SessionContextRepository

- 保存最近 5 轮业务上下文
- 关闭话题时清空上下文

### QueryLogRepository

- 只记录真正执行过的查询
- 问候、澄清、边界、结束语不写 SQL 审计

## 当前上下文模型

```json
{
  "entity_context": {
    "city": "南京市",
    "county": null,
    "sn": null
  },
  "query_frame": {
    "query_family": "anomaly",
    "intent": "soil_anomaly_query",
    "metric": null,
    "top_n": null,
    "render_mode": null,
    "audience": null
  },
  "resolved_window": {
    "start_time": "2026-04-01 00:00:00",
    "end_time": "2026-04-07 23:59:59",
    "time_label": "last_week",
    "time_explicit": true
  },
  "base_query_family": "anomaly"
}
```

## 时间实现要求

- 输入时间语义先解析为 `time_range`
- 进入 `TimeResolveService` 后统一产出 `start_time` / `end_time`
- 继承上一轮时保留绝对窗口，不重新漂移计算
- `latest_business_time` 来源于仓库查询 `MAX(create_time)`

## 模板与规则

- 规则配置来自 `metric_rule`
- 预警模板配置来自 `warning_template`
- 模板文件当前位于 `apps/agent/app/templates/soil_warning.j2`
- 模板占位符只允许当前运行时契约字段

## 测试要求

- DDL、文档、代码契约一致
- 导入链路只接受原始字段
- 任何旧字段回漂都应被硬门禁测试直接拦截
- 正式 Case 只维护 `testdata/agent/soil-moisture/case-library.md`
