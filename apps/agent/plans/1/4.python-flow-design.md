# 4. 墒情 Agent Python 工程方案（受限 Flow 单 Agent，当前实现版）

> 本文档用于承接《墒情 Agent 详细实施方案》，重点解决“如何用 Python + 自研受限 Flow 落地”。  
> 本文档是工程设计文档，不替代业务规则文档，不替代 SQL 模板文档，不替代验收测试矩阵。

---

## 1. 文档目标

本方案解决以下问题：

- 如何把当前墒情 Agent 业务方案落成 Python 工程
- 为什么当前阶段采用 **受限 Flow 单 Agent**
- 如何在 **不引入 LangChain / LangGraph** 的前提下保持可控编排
- Python 项目目录、模块边界、状态结构、节点职责如何拆分
- 如何为后续预警、专家知识增强、可选多 Agent 升级预留边界

一句话目标：

> 用自研受限 Flow 搭建一个“强约束、强事实、强规则、强模板”的 Python 单 Agent 系统，并保留未来升级空间。

---

## 2. 设计结论

### 2.1 当前推荐方案

当前推荐采用：

- **单 Agent**
- **受限 Flow 编排**
- **Pydantic 强类型 `FlowState`**
- **Service / Repository 分层**
- **SQL、规则、模板、日志全部确定性实现**
- **Redis 只负责短期上下文与缓存**
- **LLM 只负责：**
  - 意图与槽位提取
  - 受控自然语言生成

### 2.2 为什么当前不引入 LangChain / LangGraph

原因如下：

- 当前业务域仍然单一：只做墒情
- 当前主流程稳定，适合固定步骤 + 条件分支
- 数据真实性要求极高，不适合过早引入自由度更高的编排框架
- 当前最复杂的地方不是“框架能力”，而是“强规则 + 强事实 + 强门禁 + 硬降级”
- 自研受限 Flow 更容易做到：
  - 输入输出强约束
  - 路由可观察
  - 降级路径可验证
  - 多轮继承可控

### 2.3 当前方案的升级路线

系统建议按以下路径演进：

```text
阶段 1：单 Agent 主 Flow
阶段 2：单 Agent 主 Flow + 预警 / 建议增强节点
阶段 3：单 Agent 主 Flow + 专家知识增强
阶段 4：如复杂度确实提升，再评估 LangGraph 或多 Agent
```

结论：

- **现在做单 Agent 是合理的**
- **未来可升级，但当前不应绑定框架**
- 前提是现在的单 Agent 必须按“未来可拆”的方式设计

## 3. 总体工程架构

### 3.1 七层架构

系统分为 7 层：

```text
API 层
  ↓
Flow Orchestrator 层
  ↓
Flow Node 层
  ↓
Domain Service 层
  ↓
Repository 层
  ↓
LLM Adapter 层
  ↓
Schema / Contract 层
```

### 3.2 各层职责

#### 1）API 层

负责：

- 提供 HTTP 接口
- 请求入参校验
- 构造 `FlowState`
- 调用 Flow 主入口
- 返回标准响应

推荐技术：

- `FastAPI`

#### 2）Flow Orchestrator 层

负责：

- 维护 `FlowState`
- 驱动节点执行顺序
- 根据节点结果做条件跳转
- 统一终止、澄清、阻断、降级出口

推荐实现：

- 自定义 `FlowRunner`
- 显式 `route_table`
- 显式 `terminal_actions`

#### 3）Flow Node 层

负责：

- 执行单一职责节点
- 只修改自己拥有的字段
- 返回 `next_action / status / state_patch`

典型节点：

- `InputGuardNode`
- `IntentSlotExtractNode`
- `HistoryContextMergeNode`
- `TimeResolveNode`
- `RegionResolveNode`
- `ExecutionGateNode`
- `SoilDataQueryNode`
- `RuleEngineNode`
- `TemplateRenderNode`
- `AdviceComposeNode`
- `ResponseGenerateNode`
- `DataFactCheckNode`
- `AnswerVerifyNode`
- `FallbackGuardNode`

#### 4）Domain Service 层

负责：

- 业务逻辑
- 规则计算
- 模板装配
- 上下文继承
- 执行门禁
- 查询计划生成

这是整个系统最稳定、最应复用的一层。

#### 5）Repository 层

负责：

- MySQL 查询
- 参数化 SQL
- 表数据读取与写入
- 查询日志落库
- 会话上下文持久化辅助读写

这一层不承载业务判断。

#### 6）LLM Adapter 层

负责：

- 调用大模型
- 在确定性解析无法确认意图时，尝试输出结构化意图解析
- 在确定性回答已生成后，尝试输出受控润色草稿

这一层是可选增强，只做模型交互，不做业务真相判断。

#### 7）Schema / Contract 层

负责：

- 定义请求/响应 DTO
- 定义 `FlowState`
- 定义 Node I/O
- 定义错误码与标准结果结构

推荐技术：

- `Pydantic v2`

## 4. 技术栈建议

| 层级 | 推荐技术 | 说明 |
|---|---|---|
| Web API | `FastAPI` | 轻量、生态成熟、适合 Agent API |
| Flow 编排 | 自研 `FlowRunner` | 固定步骤、显式路由、便于调试 |
| 数据模型 | `Pydantic v2` | 强类型、校验清晰 |
| 配置管理 | `pydantic-settings` | 环境配置统一管理 |
| 数据访问 | `SQLAlchemy 2.x AsyncEngine` | 当前明确采用异步引擎 |
| MySQL 驱动 | `asyncmy` | 当前明确采用异步 MySQL 驱动 |
| 模板渲染 | `Jinja2` | 适合标准文本模板 |
| 日志 | `structlog` 或标准 `logging` | 推荐结构化日志 |
| 缓存 / 会话 / 短期记忆 | `Redis` | 当前固定用于最近 5 轮业务上下文与短期运行态 |
| 测试 | `pytest` | 搭配 Task16 测试矩阵 |
| LLM SDK | `DashScope / 千问 SDK` 或兼容 HTTP Client | 可选，仅用于兜底结构化提取与受控润色 |

不推荐当前直接重度引入：

- `LangChain`
- `LangGraph`
- 多 Agent Supervisor 框架
- 复杂 Workflow 持久化框架

原因：

- 当前阶段会显著增加理解成本与实现复杂度
- 你现在最重要的是把单 Agent 主流程做稳
- 自研 Flow 更容易控制事实边界、失败路径与上下文继承

## 7. `FlowState` 设计

`FlowState` 是整个系统的运行时核心。

### 7.1 设计原则

- 节点共享同一份状态对象
- 节点只能修改自己负责的字段
- 路由只依据白名单字段判断
- 不允许节点临时塞无约束字段
- 所有终止态都必须有标准响应结构

### 7.2 核心字段建议

```text
request_id
trace_id
session_id
turn_id
user_input
channel
timezone

input_type
intent
answer_type
route_target

raw_slots
merged_slots
context_used

business_time
execution_gate_result

query_plan
query_result
rule_result
template_result
advice_result

answer_bundle
query_log_entries
context_to_save
node_trace
errors
final_status
step_count
retry_count
node_visit_counts
```

### 7.3 多轮上下文策略

权威策略如下：

- 当前轮运行态权威来源：内存中的 `FlowState`
- 最近 5 轮业务上下文权威来源：`Redis SessionContextStore`
- 查询日志与调试快照权威来源：MySQL / Debug 持久化

明确约束：

- Redis 中不保存整份 `FlowState`
- Redis 只保存可继承的稳定对象：
  - `domain`
  - `region`
  - `time_window`
  - `entity_reference`
  - `last_intent`
- 不直接继承：
  - 事实结论
  - 排名结果
  - 预测结果
  - 预警等级结论

## 8. 节点协议设计

### 8.1 Node 输入输出协议

每个节点都遵守统一协议：

```text
输入：
- FlowState

输出：
- state_patch
- next_action
- node_status
- optional_debug_payload
```

### 8.2 `next_action` 建议枚举

```text
continue
safe_end
clarify_end
boundary_end
block_end
retry_response
fallback_end
verified_end
```

### 8.3 统一节点约束

- 节点内部不得直接写 HTTP 响应
- 节点内部不得直接拼 SQL
- 节点异常必须被包装成标准错误
- 节点返回必须可追踪
- 节点只能修改自己白名单内的状态字段
- 嵌套对象不得使用无约束的整对象覆盖

### 8.4 路由完整性约束

- 每个节点允许返回的 `next_action` 必须显式登记
- 每个 `next_action` 必须能路由到：
  - 下一个节点
  - 或统一终止动作
- 系统启动时必须执行 route table 校验
- 任何未登记 `next_action` 都视为设计错误，不允许运行时静默兜底

### 8.5 Fallback 统一出口约束

- 业务节点不得直接结束到 `fallback_end`
- 所有兜底路径统一走：
  - `fallback` -> `FallbackGuard` -> `fallback_end`
- `FallbackGuard` 负责：
  - 汇总错误
  - 生成最终安全回答
  - 清理半成品模板或脏草稿

### 8.6 循环与重试保护

- `FlowState` 必须记录：
  - `step_count`
  - `retry_count`
  - `node_visit_counts`
- `FlowRunner` 必须设置：
  - `max_steps`
  - `max_retry_count`
- `DataFactCheck -> ResponseGenerate` 的回路必须有限次
- 超过上限时统一进入 `FallbackGuard`

## 9. 可靠性与降级设计

### 9.1 失败优先级

建议统一为：

1. **继续执行但降能力**
2. **继续执行但降格式**
3. **继续执行但降结论**
4. **终止执行并澄清**
5. **终止执行并安全报错**

### 9.2 关键降级策略

- 模型不可用：保持确定性意图解析 + 确定性回答或固定兜底话术
- Redis 不可用：退回单轮模式
- DB 查询失败：安全报错，不编造事实
- 规则失败：只给事实，不给风险判断
- 模板失败：退回普通结构化回答
- AnswerVerify 失败：退回保守兜底答案
- 路由漏配 / 非法 `next_action`：启动即失败，不带病运行
- 节点执行超步数：直接走统一兜底

### 9.3 多轮与知识增强的额外边界

- `SessionContextStore` 只保存白名单上下文，不保存事实结论
- 同一 `session_id` 的上下文写入必须带 `turn_id`
- 旧轮次不得覆盖新轮次
- 专家知识 / 知识库只参与：
  - 原因解释
  - 处置建议
  - 背景补充
- 专家知识 / 知识库不得参与：
  - 精确计数
  - TopN 排名
  - 趋势统计
  - 预警等级判断

## 10. 对未来升级的预留边界

虽然当前不做 LangGraph / 多 Agent，但建议预留这三类边界：

- `flow/orchestrator.py`：未来可替换为更高层 supervisor
- `services/advice_service.py`：未来可独立成建议 Agent
- `repositories/session_context_repository.py`：未来可支持更复杂对话管理

前提仍然是：

- 当前阶段先把单 Agent 主 Flow 做稳
- 不为未来而引入当前用不到的抽象

---

## 11. 最终结论

- 当前阶段：**受限 Flow 单 Agent**
- 编排方式：**自研 FlowRunner + 显式 route_table**
- 状态模型：**Pydantic `FlowState`**
- 记忆策略：**Redis 短期上下文，不存整份运行态**
- 事实边界：**数据库与规则引擎说了算**
- 模型边界：**只负责理解和表达，不负责业务真相**

一句话总结：

> 对当前墒情 Agent 来说，最优解不是先引入 LangChain / LangGraph，而是做一个“未来可拆、当前足够稳”的 Python 受限 Flow 单 Agent 系统。
