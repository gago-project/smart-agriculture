# 9. 土壤墒情 Agent 地区别名补全与模糊匹配设计

## 背景

当前 `IntentSlotService` 的地区识别只做事实表全称精确命中，例如：

- `南京最近一个月的数据` 不会自动补成 `南京市`
- `如东最近怎么样` 不会自动补成 `如东县`

这会导致用户常见简称落不到地区槽位，进而回退到默认概览或错误意图。

## 目标

- 优先从现有墒情事实数据中总结出地区别名，减少人工维护
- 支持常见简称和轻度错字，例如 `南京 -> 南京市`、`如东 -> 如东县`、`南晶 -> 南京市`
- 命中多个候选时不自动猜，直接澄清，避免误查
- Qwen 返回的地区槽位也走同一套规范化逻辑

## 初始化与种子

- 新增 `region_alias` 表，承载 `alias_name -> canonical_name`
- 第一版静态种子从当前 `fact_soil_moisture` 的 `city_name / county_name / town_name` 去重汇总生成
- 生成规则：
  - 全称自映射：`南京市 -> 南京市`
  - 去行政后缀简称：`南京 -> 南京市`、`溧水 -> 溧水区`、`如东 -> 如东县`
- 生成后的静态结果写入 `infra/mysql/init/003_insert_soil_data.sql`
- 后续如发现特殊简称、口语别名或歧义，需要少量人工补充

## 解析策略

- 精确命中优先：
  - canonical 全称命中
  - alias 精确命中
- 轻度模糊兜底：
  - 仅允许短距离错字，当前实现限定为**一编辑距离**
  - 只在唯一高置信候选时自动补全
  - 只在疑似“地区短语”片段内尝试，不对整句任意位置做全局模糊替换
- 候选优先级：
  - `town > county > city`
  - 更具体地区优先于父级地区
- 例如：
  - `南京最近一个月的数据` -> `city_name=南京市`
  - `如东最近怎么样` -> `county_name=如东县`
  - `南京市溧水区最近怎么样` -> 优先 `county_name=溧水区`，并补齐 `city_name=南京市`

## 歧义处理

- 多候选时不进入 SQL 查询
- 直接返回 `clarification_needed`
- 典型场景：
  - 同一别名对应多个地区
  - 轻度模糊同时命中多个高相似候选
- 原则：宁可澄清，不可误查

## 模糊检测边界

- 当前不是通用全文检索，也不是开放式相似地名推荐
- 只做轻量业务纠错：
  - `苏洲最近一个月的数据` -> `苏州市`
  - `南通最近7天墒情怎么样` -> `南通市`
- 不允许直接把所有相似地名都自动替换
- 如果命中了多个可能地区，例如人工补充了多个 `新区`，则必须返回澄清
- 设计目标是提升简称/错字命中率，同时维持 SQL 查询前的保守性

## 与现有 Flow 的关系

- `IntentSlotService` 不再直接遍历 `known_region_names_async()` 做字符串包含判断
- 改为调用统一地区解析器
- `RegionResolveService` 继续负责“解析后槽位是否存在于事实表”的校验
- Qwen 结构化输出里的 `city_name / county_name / town_name` 也要统一走 alias 规范化

## 验收用例

- `南京最近一个月的数据`：补全为 `南京市`
- `南通最近7天墒情怎么样`：补全为 `南通市`
- `如东最近怎么样`：补全为 `如东县`
- `南晶最近一个月的数据`：唯一高置信命中时允许补全为 `南京市`
- 多候选别名：必须澄清，不直接查询
