# 1. 墒情 Agent 详细实施方案

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**目标：** 构建一个面向墒情业务的生产级 Agent，能够理解自然语言、处理非业务闲聊、安全继承最近 5 轮上下文、查询真实 MySQL 数据、基于结构化规则判断墒情状态，并按墒情预警模板输出结果，同时防止数字和事实漂移。

**架构：** 采用一个主控 `SoilMoistureChatAgent` 加一条自研受限 Flow 和一组确定性工具。`fact_soil_moisture` 是监测事实唯一来源；PDF 作为规则原始依据，当前运行时规则存放在 `metric_rule.rule_definition_json` 中，由 `SoilStatusRuleEngine` 执行；LLM 只负责理解输入、提取槽位、承接上下文、组织回答，不负责编造数字、判断等级或改写模板核心内容。

**技术栈：** 自研受限 Flow 编排、MySQL、Redis、规则 JSON、模板渲染器、可选专家知识库（仅解释和建议，不存监测事实）、回归测试集、结构化日志。当前实现不引入 `LangChain / LangGraph`，未来仅在流程复杂度明显超过受限 Flow 时再评估。

## 0. 全局设计结论

### 0.1 当前范围

- 当前业务域：`soil_moisture`
- 当前数据来源：Excel 导入 MySQL
- 后续可扩展数据来源：实时数据库 / 流式数据
- 当前上下文策略：固定保存最近 5 个业务轮次
- 当前编排策略：自研受限 Flow；固定节点、显式路由、统一降级出口
- 当前规则策略：**PDF 原文 + `metric_rule` 规则 JSON + 引擎代码**
- 当前知识库策略：可选；只用于解释与建议，不参与监测事实统计
- 当前项目已有表：
  - `fact_soil_moisture`
  - `etl_import_batch`
  - `metric_rule`
  - `admin_change_log`

### 0.2 本次交付范围

本次只做：

- 墒情查询
- 墒情规则判断
- 按墒情模板输出查询结果 / 预警结果
- 5 轮上下文
- 查询日志

本次不做：

- 虫害规则
- 气象规则
- 正式预警发布闭环
- `warning_event`
- 复杂规则拆成三张归一化规则表
- `rule_document`
- 长期会话摘要记忆

### 0.3 核心约束

- 监测数值必须来自 `SoilDataQueryTool`
- 墒情状态与预警等级必须来自 `SoilStatusRuleEngine`
- 模板文本必须来自规则模板，不允许 LLM 自由改写模板核心内容
- 最终文本中的数字、时间、地区、设备号必须通过 `DataFactChecker`
- 非业务输入不得触发 SQL
- 空数据不得回答为“正常”
- 附近点参考如果存在，必须明确标记为“参考数据，不代表目标地区真实墒情”

### 0.4 命名与格式约定

- 时间格式：`YYYY-MM-DD HH:mm:ss`
- 时区：默认 `Asia/Shanghai`
- 标识格式：
  - `record_id`：UUID
  - `batch_id`：复用现有 `etl_import_batch.batch_id`，类型为 `char(36)`
  - `query_id`：`soil_query_yyyyMMddHHmmss_xxxxxx`
  - `rule_id`：`soil_rule_xxxxxx`
  - `session_id`：`agent_session_xxxxxx`
  - `turn_id`：会话内递增整数
- 数值显示：
  - 默认保留源数据精度
  - 如需展示格式化，必须定义统一 `display_precision` 策略
  - 不允许 LLM 擅自四舍五入
- 空值处理：
  - 如 `water40cm = null`，页面和回答中应表达为“暂无该层数据”，不能写成 `0`

### 0.5 标准枚举

```text

input_type:
  greeting                寒暄问候，例如“你好”

  recent_summary          最近概览查询

  normal                  正常
  heavy_drought           重旱
  waterlogging            涝渍

  not_triggered           未达到预警条件

  unknown                 状态未知

  heavy_drought           重旱
  waterlogging            涝渍
  device_fault            设备故障
  none                    无预警等级 / 不输出等级
  unknown                 等级未知
```

补充说明：

- `input_type` 用于判断是否进入业务链路。
- `intent` 用于决定调用哪些工具。
- `answer_type` 用于规范最终回答结构。
- `query_type` 用于约束查询计划和 SQL 白名单。
- `soil_status` 是规则引擎输出的业务状态，不等同于原始监测值。
- `warning_level` 是内部等级枚举，只用于模板与规则结果传递，不等同于用户展示文案。
- `display_label` 是最终面向用户的展示文案，应由 `soil_status + warning_level + render_mode` 共同决定。
- 对“未命中预警”的场景，推荐内部返回 `soil_status = not_triggered`、`warning_level = none`，展示文案使用“未达到预警条件”，不要把三者混写。
- `tool_status` 用于描述工具调用结果，便于排障和日志分析。

### 1.4 核心 Tool I/O 契约表

本节目的：

- 给后端、Agent、测试同学一个统一的工具输入输出约束
- 避免不同模块之间“字段名看起来差不多，但实际含义不一致”
- 为后续接口实现、单测和联调提供固定边界

#### 1.4.1 工具统一返回外壳

除纯内部函数外，所有核心工具建议统一返回以下结构：

```json
{
  "tool_name": "SoilDataQueryTool",
  "status": "success",
  "error_code": null,
  "error_message": null,
  "data": {},
  "meta": {
    "request_id": "req_20260420103000_001",
    "session_id": "agent_session_abc123",
    "duration_ms": 18
  }
}
```

统一字段说明：

| 字段名 | 类型 | 说明 |
|---|---|---|
| `tool_name` | string | 当前工具名称 |
| `status` | enum | 取值：`success` / `empty` / `failed` / `blocked` |
| `error_code` | string/null | 错误码，成功时为空 |
| `error_message` | string/null | 错误说明，成功时为空 |
| `data` | object | 实际业务结果 |
| `meta.request_id` | string | 请求 ID |
| `meta.session_id` | string | 会话 ID |
| `meta.duration_ms` | int | 工具执行耗时，毫秒 |

建议标准错误码：

| 错误码 | 中文说明 |
|---|---|
| `INVALID_INPUT` | 输入参数不合法 |
| `NEED_CLARIFICATION` | 信息不足，需要用户补充 |
| `REGION_NOT_FOUND` | 未识别到有效地区 |
| `DEVICE_NOT_FOUND` | 未识别到有效设备 |
| `NO_DATA_IN_PERIOD` | 指定时间范围无数据 |
| `RULE_NOT_FOUND` | 未找到启用规则 |
| `RULE_NOT_APPLICABLE` | 当前规则不适用 |
| `TEMPLATE_FIELD_MISSING` | 模板渲染字段缺失 |
| `FACT_MISMATCH` | 文本与事实不一致 |
| `INTERNAL_ERROR` | 系统内部错误 |

#### 1.4.2 核心工具契约

| 工具 | 主要职责 | 输入字段 | 输出字段 | 空结果 / 失败约定 | 备注 |
|---|---|---|---|---|---|
| `InputGuard` | 判断是否进入业务链路 | `user_input`、`session_id`、`channel` | `input_type`、`allow_business_flow`、`reason`、`suggested_answer_type` | 闲聊或无意义输入返回 `status=blocked`，不进入 SQL | 非业务输入必须在这里被拦住 |
| `IntentSlotExtractor` | 识别意图和槽位 | `user_input`、`last_5_turn_context`、`database_latest_sample_time`（可选） | `intent`、`answer_type`、`confidence`、`slots.time_expr`、`slots.city_name`、`slots.county_name`、`slots.town_name`、`slots.device_sn`、`slots.top_n`、`slots.need_template` | 置信度不足返回 `status=blocked` + `error_code=NEED_CLARIFICATION` | 只提取，不查库 |
| `HistoryContextManager` | 继承最近 5 轮业务上下文 | `session_id`、`turn_id`、`current_slots`、`current_intent` | `merged_slots`、`inherited_fields`、`context_turns_used` | 无历史上下文返回 `status=success`，`inherited_fields=[]` | 最近 5 轮写死，运行时维护，不落表 |
| `TimeResolver` | 将相对时间转成绝对时间 | `time_expr`、`database_latest_sample_time`、`context_time_range`（可选） | `query_start_time`、`query_end_time`、`time_basis`、`resolution_mode`、`resolved_text` | 无法稳定解析时返回 `status=blocked` + `error_code=NEED_CLARIFICATION`；但应优先尝试 `SQL-07 / 场景 4` | 业务时间基准必须来自库内最新时间 |
| `RegionResolver` | 解析市、县、乡镇、设备 | `raw_region_tokens`、`raw_device_tokens`、`context_slots`、`region_alias_map` | `city_name`、`county_name`、`town_name`、`device_sn`、`resolution_source`、`confidence`、`matched_alias`、`ambiguity_candidates` | 未命中地区返回 `REGION_NOT_FOUND`；别名/轻度模糊出现多个候选时返回 `NEED_CLARIFICATION`；未命中设备返回 `DEVICE_NOT_FOUND` | 允许从上下文继承地区或设备；地区识别优先走 `region_alias` 静态映射，再做唯一高置信轻度模糊补全 |
| `ExecutionGate` | 对查询请求做执行门禁 | `query_type`、`filters`、`aggregation_level`、`limit`、`top_n` | `allow_execute`、`policy_decision`、`effective_time_range`、`effective_limit`、`degrade_strategy` | 超限返回 `status=blocked` 或自动降级 | 生产保护层，位于 `SoilDataQueryTool` 前 |
| `SoilDataQueryTool` | 执行受控 SQL 查询 | `query_type`、`filters`、`aggregation_level`、`limit`、`sql_template_id` | `rows`、`summary`、`row_count`、`source_files`、`query_sql_template_id`、`query_conditions` | 无数据返回 `status=empty`；SQL 异常返回 `failed` | 禁止直接执行 LLM 自由生成 SQL |
| `SoilStatusRuleEngine` | 根据规则判断状态和等级 | `rule_code`、`fact_rows` 或 `aggregated_facts`、`business_time` | `soil_status`、`warning_level`、`matched_rule_ids`、`logic_basis`、`triggered` | 规则不存在返回 `RULE_NOT_FOUND`；规则不适用返回 `RULE_NOT_APPLICABLE` | 阈值不得写死在 Prompt 中 |
| `TemplateRenderer` | 按模板渲染固定文案 | `template_code`、`template_context`、`render_mode` | `rendered_text`、`used_fields`、`missing_fields` | 缺字段返回 `blocked` + `TEMPLATE_FIELD_MISSING` | 模板核心内容不允许被 LLM 重写 |
| `KnowledgeRetriever` | 提供解释和建议素材 | `question`、`fact_context`、`audience_type` | `knowledge_items`、`sources`、`retrieval_mode` | 没有命中文档时返回 `empty`，但不影响查询主流程 | 只用于建议解释，不参与事实统计 |
| `ResponseGenerator` | 组织自然语言回答 | `answer_type`、`facts`、`rule_result`、`template_result`、`knowledge_items` | `draft_answer`、`answer_sections`、`used_evidence` | 缺核心事实时返回 `blocked` | 不允许擅自改写数值、时间、设备号 |
| `DataFactChecker` | 校验文本与事实一致性 | `draft_answer`、`fact_bundle`、`rule_result` | `checked_answer`、`pass`、`mismatch_list` | 不一致返回 `failed` + `FACT_MISMATCH` | 重点检查数字、时间、地区、SN 编号 |
| `AnswerVerifier` | 最终完整性与边界校验 | `checked_answer`、`answer_type`、`required_fields` | `final_answer`、`verify_pass`、`issues` | 校验不通过返回 `blocked` | 防止答非所问、漏时间范围、漏数据来源 |

## 2. 现有表与本次方案对齐

### 2.1 对齐结论

当前项目无需推倒重来，主干表直接复用：

| 现有表 | 在方案中的角色 | 本次动作 |
|---|---|---|
| `fact_soil_moisture` | 墒情事实表 | 直接复用 |
| `etl_import_batch` | 导入批次表 | 直接复用 |
| `metric_rule` | 运行时规则表 | 直接复用，并新增 `soil_warning_v1` |
| `admin_change_log` | 管理操作审计 | 直接复用 |

本次新增：

| 新表 | 作用 |
|---|---|
| `agent_query_log` | 记录系统实际查了什么数据 |

本次不新增：

| 不做项 | 原因 |
|---|---|
| `warning_event` | 本次只做墒情查询/规则判断/模板输出，不做正式预警发布闭环 |
| `soil_rule_set / soil_rule_item / soil_rule_condition` | 当前规则很少，用 `metric_rule.rule_definition_json` 更合适 |
| `rule_document` | 当前墒情规则较少且基本固定，直接维护在 `metric_rule` 中即可 |

| `agent_turn_log` | 当前阶段不记录完整问答日志，避免表过重；查询审计由 `agent_query_log` 承担 |

### 2.2 数据库索引建议

设计原则：

- 先满足当前 MVP 查询主路径，不一次性加过多索引
- 优先支持高频查询维度：`sample_time`、`device_sn`、`city_name`、`county_name`
- 兼顾异常查询、排名查询和日志追溯

## 3. 回答分类设计（10 类）

当前方案正式定义 **10 类回答类型**，分为：

- **6 类业务回答**
- **4 类控制/兜底回答**

这样设计的原因是：

- 能覆盖墒情查询、异常分析、模板输出、解释建议
- 能避免“你好”“看看”“没数据”“问题不清楚”时答非所问
- 便于后续对回答结构、工具调用、验收测试进行约束

### 3.1 业务回答类（6 类）

#### 1）`soil_summary_answer`：墒情概览回答

适用问题：

- 最近墒情怎么样？
- 帮我看看最近有没有问题。
- 这批数据整体情况如何？

必须调用：

- `TimeResolver`
- `SoilDataQueryTool`
- `SoilStatusRuleEngine`（可选，若需要状态汇总）

回答内容：

- 时间范围
- 数据范围
- 总体情况
- 异常数量
- 重点地区/设备
- 数据来源

示例：

```text
根据库内最新监测时间 2025 年 6 月 1 日 00 时，按最近 7 天数据统计，当前共查询到 X 条墒情记录，其中 Y 条存在异常，主要集中在 XX 县、XX 镇。数据来源为本次导入的土壤墒情仪数据.xlsx。
```

#### 2）`soil_ranking_answer`：排名/对比回答

适用问题：

- 过去一个月哪里最严重？
- 哪个县墒情最差？
- 哪些设备异常分最高？

必须调用：

- `TimeResolver`
- `RegionResolver`（可选）
- `SoilDataQueryTool`

回答内容：

- 排名口径
- 时间范围
- Top N 结果
- 排序字段
- 数据来源

说明：

- “最严重”默认按 `soil_anomaly_score`
- 如果用户问“最旱”，优先走规则状态，其次看 `water20cm`

#### 3）`soil_detail_answer`：区域/设备详情回答

适用问题：

- 如东县最近怎么样？
- SNS00204333 最近有没有异常？
- 这个设备上周数据如何？

必须调用：

- `HistoryContext Manager`（运行时，可选）
- `TimeResolver`
- `RegionResolver`
- `SoilDataQueryTool`
- `SoilStatusRuleEngine`

回答内容：

- 地区或设备
- 最新监测时间
- 各层含水量
- 状态判断
- 趋势简述
- 数据来源

示例：

```text
设备 SNS00204333 最新监测时间为 2025 年 6 月 1 日 00 时，20cm 相对含水量为 83.18%，当前规则判定状态为未达到预警条件。数据来源：土壤墒情仪数据.xlsx。
```

#### 4）`soil_anomaly_answer`：异常/状态分析回答

适用问题：

- 有没有异常？
- 哪些地方需要关注？
- 有没有重旱？
- 有没有涝渍？

必须调用：

- `TimeResolver`
- `SoilDataQueryTool`
- `SoilStatusRuleEngine`

回答内容：

- 异常数量
- 异常类型
- 触发依据
- 涉及地区/设备
- 是否建议进入模板输出

示例：

```text
指定时间范围内共发现 X 条需要关注记录，其中 A 条为重旱、B 条为涝渍。状态判断来自墒情规则引擎，不是模型自行判断。
```

#### 5）`soil_warning_answer`：预警模板回答

适用问题：

- 生成一条墒情预警。
- 这个设备需要发预警吗？
- 按模板输出。

必须调用：

- `SoilDataQueryTool`
- `SoilStatusRuleEngine`
- `TemplateRenderer`
- `DataFactChecker`

回答内容：

- 是否触发预警
- 预警等级
- 触发规则
- 标准模板文案
- 数据来源

未触发时示例：

```text
该设备最新记录未满足墒情预警规则，如需按模板展示当前监测结果，可标注为“未达到预警条件”。
```

触发时示例：

```text
2025 年 12 月 24 日 00 时淮安市盱眙县 SN 编号 SNS00214018 土壤墒情仪监测到相对含水量 15.96%，预警等级 重旱，请大田/设施大棚/林果相关主体关注！
```

#### 6）`soil_advice_answer`：解释/指导建议回答

适用问题：

- 83.18 是什么意思？
- 这种情况应该怎么办？
- 农户要注意什么？
- 大棚要怎么处理？

必须调用：

- `SoilDataQueryTool`（可选）
- `SoilStatusRuleEngine`（可选）
- `KnowledgeRetriever`

回答内容：

- 数据事实
- 指标解释
- 建议内容
- 适用对象
- 不确定性说明

建议回答结构：

```text
数据事实：
……

指标解释：
……

建议：
……

说明：
由于当前缺少作物类型、土壤类型和未来天气数据，以上建议仅作为一般性管理参考。
```

### 3.2 控制/兜底回答类（4 类）

#### 7）`safe_hint_answer`：非业务安全提示回答

适用问题：

- 你好
- 在吗
- 你是谁
- 你能做什么
- 哈哈哈
- ？？？

不调用业务工具。

回答内容：

- 简短回应
- 说明能力范围
- 给出示例问题

示例：

```text
你好，我是墒情智能助手，可以帮你查询墒情数据、分析异常、判断预警并按标准模板输出结果。你可以问我：最近墒情怎么样？过去一个月哪里最严重？SNS00204333 有没有异常？
```

#### 8）`clarification_answer`：澄清/选项引导回答

适用问题：

- 看看
- 查一下
- 情况
- 帮我看一下

特点：

- 输入太短
- 无法判断具体业务意图
- 不应直接查库

回答内容：

```text
你想查看哪类墒情信息？可以选择：
1. 最近墒情概况
2. 异常点位
3. 某个地区情况
4. 某个设备情况
5. 按模板输出结果
```

#### 9）`fallback_answer`：无数据/找不到/不可判断回答

适用场景：

- 地区不存在
- 设备不存在
- 指定时间无数据
- 规则无法判断
- 模板缺字段

必须做到：

- 不编结果
- 不说正常
- 给下一步选择

示例：

```text
如东县在指定时间范围内暂无墒情数据。当前库内最新监测时间为 2025 年 6 月 1 日 00 时。是否改查最新数据？
```

或：

```text
未在当前墒情数据中找到“XX乡镇”。你可以检查名称是否准确，或改查已有地区。
```

#### 10）`boundary_answer`：超出能力边界回答

适用问题：

- 帮我写首诗
- 查一下股票
- 明天会不会下雨
- 预测今年所有虫害

回答内容：

- 说明当前只支持墒情相关能力
- 给出可问问题

示例：

```text
我当前主要支持墒情相关业务，包括数据查询、异常分析、规则判断和模板输出。你可以问：最近墒情怎么样？或者某个设备是否异常。
```

### 3.4 回答类型对应的 SQL 查询模板

说明：

- 本节只给出 **受控 SQL 模板**，用于指导后端实现 `SoilDataQueryTool`
- **SQL 只负责查询事实，不负责判定预警等级**
- 墒情状态、预警等级、模板输出仍由：
  - `SoilStatusRuleEngine`
  - `TemplateRenderer`
  负责
- 所有 SQL 都应通过参数化方式执行，禁止直接拼接用户输入

#### 3.4.-1 查询执行门禁（ExecutionGate / QueryPolicy）

`ExecutionGate` 位于 `SoilDataQueryTool` 前，用于保护数据库与 Agent 输出质量。

建议固定以下门禁策略：

| 策略项 | 建议值 | 说明 |
|---|---|---|
| `max_time_range_days_summary` | 90 | 概览类默认最多查近 90 天 |
| `max_time_range_days_ranking` | 365 | 排名类最多查近 1 年 |
| `max_time_range_days_detail` | 90 | 详情与趋势默认最多查近 90 天 |
| `max_time_range_days_anomaly` | 180 | 异常分析默认最多查近 180 天 |
| `max_limit_default` | 10 | 默认返回条数 |
| `max_limit_hard` | 100 | 硬上限，超过则阻断或降级 |
| `max_top_n` | 20 | Top N 最大值 |
| `max_group_by_cardinality` | 500 | 允许的最大聚合分组数 |
| `allow_full_table_scan_province` | false | 不允许无约束全省全表扫描 |
| `allow_batch_device_trend` | false | 不允许一次查大量设备趋势 |
| `require_filter_for_trend_query` | true | 趋势类必须带设备或地区 |
| `overflow_strategy` | `clarify` | 超限时优先让用户缩小范围 |

执行原则：

- 用户请求超过时间窗时，优先触发澄清，而不是直接放大扫描范围。
- 若用户问“全省近一年每天趋势”“过去三年所有设备排名”，应由 `ExecutionGate` 阻断或降级。
- 排名查询、趋势查询、异常查询都必须受最大时间窗和最大返回量限制。
- 允许的降级策略包括：
  - `clarify`：提示用户缩小时间范围或对象范围
  - `shrink_range`：自动收缩到可执行范围，并明确告知用户
  - `return_summary_only`：仅返回概览，不返回明细列表

ExecutionGate 推荐参数对象：

```json
{
  "policy_code": "soil_query_policy_v1",
  "max_time_range_days": {
    "recent_summary": 90,
    "severity_ranking": 365,
    "region_detail": 90,
    "device_detail": 90,
    "anomaly_list": 180,
    "latest_record": 365,
    "device_timeseries": 90,
    "region_timeseries": 90
  },
  "limit_policy": {
    "default_limit": 10,
    "max_top_n": 20,
    "max_return_rows": 100
  },
  "scan_policy": {
    "allow_unfiltered_province_scan": false,
    "allow_full_table_scan": false,
    "require_region_or_device_for_timeseries": true,
    "allow_batch_device_timeseries": false,
    "max_group_by_cardinality": 500
  },
  "overflow_policy": {
    "default_strategy": "clarify",
    "allowed_strategies": [
      "clarify",
      "shrink_range",
      "return_summary_only"
    ]
  }
}
```

ExecutionGate 推荐返回结构：

```json
{
  "tool_name": "ExecutionGate",
  "status": "blocked",
  "error_code": "QUERY_POLICY_EXCEEDED",
  "error_message": "查询范围超过系统允许上限",
  "data": {
    "allow_execute": false,
    "policy_decision": "clarify",
    "violations": [
      {
        "field": "time_range",
        "reason": "requested_days_exceed_limit",
        "requested_value": 1095,
        "allowed_value": 365
      }
    ],
    "suggested_filters": {
      "max_time_range_days": 365,
      "need_region_or_device": true,
      "max_top_n": 20
    },
    "suggested_answer_type": "clarification_answer"
  }
}
```

超限提示文案示例：

| 用户问题 | 命中门禁 | 建议回复 |
|---|---|---|
| 全省近三年所有设备排名 | 时间窗超限 + 全省大范围排名 | 当前查询范围较大，可能影响系统性能。请将时间范围缩小到近 1 年内，或指定某个市/县区后再查询。 |
| 全省近一年每天趋势 | 趋势查询缺少地区或设备约束 | 趋势查询需要指定地区或设备。你可以问“南通市近一年墒情趋势”或“SNS00204333 近 90 天趋势”。 |
| 查所有设备最近 90 天趋势 | 批量设备趋势不允许 | 当前不支持一次性返回所有设备趋势。你可以指定一个设备 SN，或改查某个地区的概览趋势。 |
| 给我前 100 个最严重设备 | `top_n` 超过上限 | 当前最多支持 Top 20 排名。是否为你查询前 20 个最严重设备？ |
| 查过去 5 年异常点位 | 异常查询时间窗超限 | 异常分析当前最多支持近 180 天。请缩小时间范围，或改查近 180 天异常情况。 |

与回答类型的关系：

- 若门禁返回 `clarify`，最终回答类型应为 `clarification_answer`
- 若门禁返回 `return_summary_only`，最终回答类型可降级为 `soil_summary_answer`
- 若门禁返回 `shrink_range`，回答中必须明确说明“已按系统上限自动缩小查询范围”
- 若门禁返回 `blocked`，不得继续调用 `SoilDataQueryTool`

#### 3.4.0 10 类回答类型与 SQL 模板对应关系

| 回答类型 | 是否需要 SQL | SQL 模板编号 | 说明 |
|---|---|---|---|
| `soil_summary_answer` | 是 | `SQL-01` | 查询整体概况、总记录数、异常数、最新时间 |
| `soil_ranking_answer` | 是 | `SQL-02` | 查询地区或设备排名 |
| `soil_detail_answer` | 是 | `SQL-03` | 查询地区/设备详情及趋势 |
| `soil_anomaly_answer` | 是 | `SQL-04` | 查询异常记录和异常类型统计 |
| `soil_warning_answer` | 是 | `SQL-05` | 查询规则判断和模板输出所需事实字段 |
| `soil_advice_answer` | 是 | `SQL-06` | 查询建议生成前的事实背景 |
| `safe_hint_answer` | 否 | 无 | 非业务安全提示，不查库 |
| `clarification_answer` | 否 | 无 | 澄清/引导，不查库 |
| `fallback_answer` | 是 | `SQL-07` | 做存在性检查、空数据检查、最新时间检查 |
| `boundary_answer` | 否 | 无 | 超出能力边界说明，不查库 |

补充说明：

- `SQL-01 ~ SQL-07` 是当前文档内部使用的受控 SQL 模板编号，便于商务、产品、后端对齐。
- 没有 SQL 模板的回答类型，不应触发事实查询。
- `soil_warning_answer` 虽然有 SQL 模板，但 SQL 只负责取事实，不负责判定预警等级。

#### 3.4.1 `SQL-01` / `soil_summary_answer` 对应 SQL

查询目标：

- 查询某个时间范围内的墒情整体概况
- 统计总记录数、最新监测时间、异常数量

输入参数：

- `start_time`
- `end_time`
- `city_name`（可选）
- `county_name`（可选）
- `device_sn`（可选）

SQL 模板（候选筛选，不作为最终异常口径）：

```sql
SELECT
  COUNT(*) AS total_record_count,
  MAX(sample_time) AS latest_sample_time,
  MIN(sample_time) AS earliest_sample_time,
  COUNT(DISTINCT device_sn) AS device_count,
  COUNT(DISTINCT county_name) AS county_count,
  COUNT(DISTINCT county_name) AS county_coverage_count,
  SUM(
    CASE
      WHEN soil_anomaly_type IS NOT NULL
       AND soil_anomaly_type <> 'normal'
      THEN 1 ELSE 0
    END
  ) AS anomaly_record_count,
  MAX(source_file) AS latest_source_file
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?)
  AND (? IS NULL OR device_sn = ?);
```

可选补充 SQL：查询异常较多地区 Top 5

```sql
SELECT
  city_name,
  county_name,
  COUNT(*) AS record_count,
  SUM(
    CASE
      WHEN soil_anomaly_type IS NOT NULL
       AND soil_anomaly_type <> 'normal'
      THEN 1 ELSE 0
    END
  ) AS anomaly_count,
  MAX(sample_time) AS latest_sample_time
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
GROUP BY city_name, county_name
ORDER BY anomaly_count DESC, latest_sample_time DESC
LIMIT 5;
```

返回字段：

- `total_record_count`
- `latest_sample_time`
- `earliest_sample_time`
- `device_count`
- `county_count`
- `county_coverage_count`
- `anomaly_record_count`
- `latest_source_file`

边界说明：

- 若 `soil_anomaly_type` 为空或未维护，可将异常数量统计降级为基于规则引擎二次判断
- 建议概览回答优先使用 `county_coverage_count` 对外表述“本次统计覆盖了多少个县区”

#### 3.4.2 `SQL-02` / `soil_ranking_answer` 对应 SQL

查询目标：

- 对地区或设备进行严重程度排序

输入参数：

- `start_time`
- `end_time`
- `city_name`（可选）
- `county_name`（可选）
- `aggregation_level`：`province` / `city` / `county` / `device`
- `ranking_scope`：`county` 或 `device`
- `limit`
- `top_n`（可由槽位提取器从“前 3 名”“前 10 个”“哪些地方”中识别）

默认口径：

- “最严重”默认按 `soil_anomaly_score` 降序
- 当前 MVP 默认按 `county` 级输出地区排名
- 如果用户明确问“全省哪里最严重”，默认返回县级排名；后续可根据 `aggregation_level` 切换为市级聚合排名
- 如果用户明确问“哪个市最严重”，则应走 `city` 级聚合
- `limit` 默认建议取 `5`
- 若用户明确指定 `Top N`，则按用户指定值执行
- 若用户未指定且使用“哪些地方”“哪些设备”这类开放式问法，建议默认放宽到 `10`
- 为防止返回过多结果，建议最大 `limit = 20`

多层级聚合建议：

- `province`：全省整体概览，一般不做多行排名，更多用于“全省当前总体最严重程度”汇总
- `city`：按设区市聚合，适合回答“哪个市最严重”
- `county`：按县区聚合，适合回答“全省哪里最严重”“哪个县最严重”
- `device`：按设备聚合，适合回答“哪个设备最严重”

按县区排名 SQL：

```sql
SELECT
  city_name,
  county_name,
  MAX(soil_anomaly_score) AS max_anomaly_score,
  AVG(soil_anomaly_score) AS avg_anomaly_score,
  AVG(water20cm) AS avg_water20cm,
  MIN(water20cm) AS min_water20cm,
  MAX(water20cm) AS max_water20cm,
  COUNT(*) AS record_count,
  MAX(sample_time) AS latest_sample_time
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?)
GROUP BY city_name, county_name
ORDER BY max_anomaly_score DESC, avg_anomaly_score DESC, latest_sample_time DESC
LIMIT ?;
```

按市级排名 SQL：

```sql
SELECT
  city_name,
  MAX(soil_anomaly_score) AS max_anomaly_score,
  AVG(soil_anomaly_score) AS avg_anomaly_score,
  AVG(water20cm) AS avg_water20cm,
  MIN(water20cm) AS min_water20cm,
  MAX(water20cm) AS max_water20cm,
  COUNT(*) AS record_count,
  COUNT(DISTINCT county_name) AS county_coverage_count,
  MAX(sample_time) AS latest_sample_time
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
GROUP BY city_name
ORDER BY max_anomaly_score DESC, avg_anomaly_score DESC, latest_sample_time DESC
LIMIT ?;
```

按设备排名 SQL：

```sql
SELECT
  device_sn,
  city_name,
  county_name,
  MAX(soil_anomaly_score) AS max_anomaly_score,
  AVG(soil_anomaly_score) AS avg_anomaly_score,
  MAX(water20cm) AS max_water20cm,
  MIN(water20cm) AS min_water20cm,
  MAX(sample_time) AS latest_sample_time
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?)
GROUP BY device_sn, city_name, county_name
ORDER BY max_anomaly_score DESC, latest_sample_time DESC
LIMIT ?;
```

返回字段：

- `city_name`
- `county_name`
- `device_sn`（设备排名时）
- `max_anomaly_score`
- `avg_anomaly_score`
- `avg_water20cm`
- `min_water20cm`
- `max_water20cm`
- `record_count`
- `county_coverage_count`（市级排名时）
- `latest_sample_time`

边界说明：

- 如果用户问“最旱”，SQL 只能查事实值，最终是否属于“重旱”必须由规则引擎判断
- 若未来数据量明显增长，建议增加市级聚合视图，例如 `vw_soil_ranking_city_daily`
- 该视图建议按“日期 + 市”提前聚合 `max_anomaly_score / avg_anomaly_score / avg_water20cm / min_water20cm / max_water20cm / county_coverage_count`
- MVP 阶段可先直接查事实表，待数据规模增大后再引入聚合视图或物化汇总表

#### 3.4.4 `SQL-04` / `soil_anomaly_answer` 对应 SQL

查询目标：

- 查询某时间范围内需要关注的异常记录

输入参数：

- `start_time`
- `end_time`
- `city_name`（可选）
- `county_name`（可选）
- `device_sn`（可选）

SQL 模板：

```sql
SELECT
  record_id,
  device_sn,
  city_name,
  county_name,
  town_name,
  sample_time,
  water20cm,
  soil_anomaly_type,
  soil_anomaly_score,

  AND (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?)
  AND (? IS NULL OR device_sn = ?)
  AND (
    (soil_anomaly_type IS NOT NULL AND soil_anomaly_type <> 'normal')
    OR (soil_anomaly_score IS NOT NULL AND soil_anomaly_score > 0)
  )
ORDER BY sample_time DESC, soil_anomaly_score DESC
LIMIT 100;
```

统计 SQL（候选分布统计，不作为最终对外异常统计）：

```sql
SELECT
  soil_anomaly_type,
  COUNT(*) AS anomaly_count
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?)
  AND (
    (soil_anomaly_type IS NOT NULL AND soil_anomaly_type <> 'normal')
    OR (soil_anomaly_score IS NOT NULL AND soil_anomaly_score > 0)
  )
GROUP BY soil_anomaly_type
ORDER BY anomaly_count DESC;
```

返回字段：

- 异常记录列表
- 异常类型分布
- `water20cm` 必须作为异常回答的原始事实字段返回，不能省略

边界说明：

- 若要回答“有没有重旱/涝渍”，最终仍应以规则引擎结果为准
- 即使 `soil_anomaly_type` 已有派生值，对外回答仍建议同步展示原始 `water20cm`，以支撑“异常到什么程度”的解释
- 本节 SQL 仅负责拉取“异常候选记录”和“候选分布”
- 正式对外异常数量、异常类型分布、最终异常状态，应以 `SoilStatusRuleEngine` 对候选记录批量复判后的结果为准
- 不允许同时存在“SQL 异常定义”和“规则引擎异常定义”两套对外口径

#### 3.4.7 `SQL-07` / `fallback_answer` 对应 SQL

查询目标：

- 做存在性检查、空数据检查、最新时间检查

场景 1：地区是否存在

```sql
SELECT
  COUNT(*) AS region_record_count
FROM fact_soil_moisture
WHERE (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?);
```

场景 2：设备是否存在

```sql
SELECT
  COUNT(*) AS device_record_count
FROM fact_soil_moisture
WHERE device_sn = ?;
```

场景 3：某时间段是否有数据

```sql
SELECT
  COUNT(*) AS period_record_count,
  MAX(sample_time) AS latest_sample_time
FROM fact_soil_moisture
WHERE sample_time BETWEEN ? AND ?
  AND (? IS NULL OR city_name = ?)
  AND (? IS NULL OR county_name = ?)
  AND (? IS NULL OR device_sn = ?);
```

场景 4：查询库内最新业务时间

```sql
SELECT
  MAX(sample_time) AS latest_sample_time
FROM fact_soil_moisture;
```

返回字段：

- `region_record_count`
- `device_record_count`
- `period_record_count`
- `latest_sample_time`

边界说明：

- `fallback_answer` 允许使用 SQL 做存在性和空结果判断
- 但不应在无数据场景下生成任何“正常”或“安全”结论
- `SQL-07` 不只是兜底查无数据，也承担“业务时间感知”职责
- 当用户说“现在的墒情”“当前情况”“最新情况”时，应优先查询场景 4，拿到库内最新业务时间，再回填给 `TimeResolver`
- 当 `TimeResolver` 无法把相对时间稳定解析为绝对时间时，也应先调用场景 4，再决定是否按“最近 7 天”或“最新一次记录”继续执行

#### 3.4.8 三类不需要 SQL 的回答

以下回答类型默认不查库：

1. `safe_hint_answer`
2. `clarification_answer`
3. `boundary_answer`

原因：

- 这三类回答本质上是输入控制和边界说明
- 如果查库，反而容易造成误触发和答非所问

#### 3.4.9 SQL 使用总原则

必须在文档中明确以下原则：

1. **SQL 只查事实，不判等级**
2. **规则引擎负责判定 `soil_status` 和 `warning_level`**
3. **模板渲染器负责输出标准模板**
4. **非业务回答不查库**
5. **所有 SQL 必须参数化执行**
6. **所有 SQL 必须由后端白名单查询计划生成，不能直接接受 LLM 输出 SQL**

### 5.1.1 规则优先级与冲突语义

必须明确以下执行语义：

- `priority` 数值越小，优先级越高
- 同一条记录允许“多规则命中候选”，但最终对外 `soil_status` 必须唯一
- 最终状态选择规则：按 `priority` 从小到大排序后，取第一个“有效命中且未被禁用窗拦截”的结果
- 若命中规则条件，但该规则当前落在 `disabled_windows` 中，则应记录为：
  - `matched_but_suppressed = true`
  - `suppressed_reason = disabled_window`
  - 但不作为最终对外状态输出
- 若所有规则都未形成有效命中，则输出：
  - `soil_status = not_triggered`
  - `warning_level = none`
- 同一输入的最终 `soil_status` 和 `warning_level` 必须稳定唯一，不能多值并存

## 7. 详细任务拆分

### Task 1：对齐事实表与导入批次

**优先级：** P0  
**负责人：** 后端 / 数据

**目标：**
- 确认 `fact_soil_moisture` 与 `etl_import_batch` 足以支撑墒情查询
- 明确原始字段、派生字段、来源追溯字段
- 明确 `fact_soil_moisture.batch_id` 与 `etl_import_batch.batch_id` 的关联关系

**输入：**
- 原始 Excel：`土壤墒情仪数据.xlsx`
- 当前数据库表结构

**输出：**
- 字段映射表
- 导入链路说明
- 查询口径说明

**验收标准：**
- 能追溯一条监测记录来自哪个文件
- 能追溯一条监测记录来自哪个导入批次
- 能拿到库内最新 `sample_time`
- 能按设备、地区、时间查询

### Task 2：InputGuard 非业务输入识别

**优先级：** P0  
**负责人：** Agent / 后端

**目标：**
- 在进入业务链路前识别“你好”“看看”“写首诗”这类输入

**输入分类：**
- `greeting`
- `capability_question`
- `meaningless_input`
- `ambiguous_low_confidence`
- `out_of_domain`
- `business_colloquial`
- `business_direct`

**验收标准：**
- “你好”不查库
- “看看”返回澄清选项
- “最近墒情怎么样”进入业务链路

### Task 3：HistoryContext Manager（固定 5 轮）

**优先级：** P0  
**负责人：** Agent / 后端

**目标：**
- 支持多轮追问、省略表达和槽位继承

**规则：**
- 仅保存最近 5 个业务轮次
- 非业务输入不写入业务上下文
- 仅允许从白名单槽位继承：`city_name`、`county_name`、`town_name`、`device_sn`、`metric`、`depth`、`resolved_time_range`
- `top_n`、排序方向、模板输出意图、建议需求等字段不得跨轮强继承
- 最近 `1~2` 轮高置信继承；第 `3~5` 轮仅弱参考，必要时降级澄清
- 支持：
  - “那上周的呢”
  - “按 20cm 看”
  - “换成如东县”

**验收标准：**
- 连续 5 轮追问能正确继承地区、设备、时间、指标
- 不会把过早轮次的 `top_n`、模板需求、隐式结论错误继承到当前轮

### Task 4：Intent & Slot 识别

**优先级：** P0  
**负责人：** Agent

**目标：**
- 识别墒情查询、设备查询、异常查询、模板输出等意图

**支持意图：**
- `soil_recent_summary`
- `soil_severity_ranking`
- `soil_region_query`
- `soil_device_query`
- `soil_anomaly_query`
- `soil_warning_generation`
- `soil_metric_explanation`
- `soil_management_advice`

**验收标准：**
- 能正确提取时间、地区、设备、指标、模板需求

### Task 5：TimeResolver

**优先级：** P0  
**负责人：** 后端 / Agent

**目标：**
- 将“最近”“上周”“过去一个月”“这一批”解析成绝对时间范围

**规则：**
- 以 `fact_soil_moisture.MAX(sample_time)` 为业务时间基准
- 不以系统当前时间作为业务最新时间
- 当用户说“现在”“当前”“最新”时，应先查询 `SQL-07 / 场景 4` 获取库内最新业务时间
- 当相对时间表达无法稳定落到绝对区间时，也应先查询 `SQL-07 / 场景 4` 再做时间解析
- 若最终仍无法确定用户意图，应降级为：
  - 优先提示“当前库内最新监测时间截至 XX”
  - 再询问用户是要查“最新一次记录”还是“最近 7 天概况”

**验收标准：**
- “最近”按库内最新时间向前 7 天
- “这一批”可基于 `etl_import_batch` + 事实表推导
- “现在的墒情”不会按系统挂钟时间硬查当天，而是会自动落到库内最新业务时间

### Task 6：RegionResolver

**优先级：** P0  
**负责人：** 后端 / Agent

**目标：**
- 解析市、县、设备 SN

**数据来源：**
- `fact_soil_moisture`
- `region_alias`（由事实表去重结果生成的静态别名种子，包含全称自映射和去行政后缀简称）
- 可选 `soil_device_dimension`

**规则：**
- 先做 canonical 全称命中和 alias 精确命中，例如 `南京 -> 南京市`、`如东 -> 如东县`
- 仅允许轻度模糊兜底，限定为地区候选短语上的唯一高置信一编辑距离命中，例如 `苏洲 -> 苏州市`
- 多候选 alias 或多候选模糊命中时，不自动猜测，直接进入 `clarification_needed`
- 命中县区或乡镇时需要补齐父级地区，例如 `如东 -> county_name=如东县, city_name=南通市`

**验收标准：**
- 能识别 `南通市`、`如东县`、`SNS00204333`
- 能将 `南京最近一个月的数据` 规范成 `city_name=南京市`
- 能将 `如东最近怎么样` 规范成 `county_name=如东县`
- 能对唯一高置信错字做轻度补全，例如 `苏洲最近一个月的数据 -> city_name=苏州市`
- 对 `新区最近怎么样` 这类多候选场景必须澄清，不能直接查库

**优先级：** P0  
**负责人：** 后端

**目标：**

- 概览
- 排名
- 区域详情
- 设备详情
- 异常清单
- 最新记录

**执行门禁：**

- 必须限制最大时间窗、最大返回量、最大 `topN`
- 不允许无约束全省全表扫描
- 趋势查询必须带地区或设备约束
- 超限时优先返回澄清或范围缩小提示

**日志要求：**
- 每次查询写入 `agent_query_log`
- 查询日志同时保留 `sql_fingerprint`（模板标识）和 `executed_sql_text`（本次可执行 SQL 文本）

**验收标准：**
- 返回结构化 JSON

### 8.3 当前方案最终结论

本次墒情 Agent 采用以下落地策略：

- 事实数据：`fact_soil_moisture`
- 导入批次：`etl_import_batch`
- 规则执行：`metric_rule` 中的 `soil_warning_v1`
- 查询日志：`agent_query_log`
- 运行时上下文：应用内存 / 缓存 / 会话对象中维护最近 5 个业务轮次
- 不做：`warning_event`、`rule_document`、`agent_turn_context`、`agent_turn_log`
