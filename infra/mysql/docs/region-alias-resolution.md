# 地区别名解析与 `region_alias` 使用设计

## 背景

数据库里保存的是规范地区名，但用户输入往往是简称、口语或轻度错字。例如：

- `南京最近一个月的数据`
- `如东最近怎么样`
- `南晶最近一个月的数据`

如果只依赖 `fact_soil_moisture` 中的全称精确命中，很多自然输入无法落到正确槽位。因此需要一套围绕 `region_alias` 的别名解析设计。

## 设计目标

- 优先复用现有事实数据自动生成地区别名，减少人工维护量。
- 支持常见简称补全，例如 `南京 -> 南京市`、`如东 -> 如东县`。
- 支持唯一高置信的轻度错字纠正，例如 `南晶 -> 南京市`。
- 一旦存在多个候选，不自动猜，必须进入澄清。
- Qwen 或其他结构化槽位输出也复用同一套规范化逻辑。

## 相关表

- [`fact_soil_moisture`](./fact_soil_moisture.md)：事实地区来源。
- [`region_alias`](./region_alias.md)：别名映射承载表。

## 初始化与种子来源

第一版别名种子从当前事实表中的 `city_name / county_name / town_name` 去重生成，生成规则包括：

- 全称自映射：`南京市 -> 南京市`
- 去行政后缀简称：`南京 -> 南京市`、`如东 -> 如东县`
- 允许保留父级信息：例如区县记录可带上所属 `parent_city_name`

生成后的结果作为当前静态种子写入初始化 SQL，后续特殊简称或歧义别名再人工补充。

生成路径：

- 代码生成：`apps/web/lib/server/regionAliasSeed.mjs`
- 命令脚本：`apps/web/scripts/generate-region-alias-seed.mjs`
- 初始化落库：`infra/mysql/init/003_insert_soil_data.sql`

## 解析策略

### 1. 精确命中优先

优先顺序：

1. 规范全称命中
2. 别名精确命中

如果一句话里能命中多个层级，则优先更具体的层级：

- `town > county > city`

例如：

- `南京最近一个月的数据` -> `city_name = 南京市`
- `如东最近怎么样` -> `county_name = 如东县`
- `南京市溧水区最近怎么样` -> 优先解析出 `county_name = 溧水区`，并补齐 `city_name = 南京市`

### 2. 轻度模糊兜底

只在以下前提下允许自动纠正：

- 只做短距离轻度错字；
- 当前实现限定为一编辑距离；
- 只在唯一高置信候选下自动补全；
- 只针对疑似地区短语片段，不做整句任意位置全文替换。

例如：

- `南晶最近一个月的数据` -> 唯一命中时补全为 `南京市`
- `苏洲最近一个月的数据` -> 唯一命中时补全为 `苏州市`

## 歧义处理

多候选场景必须进入澄清，不允许直接查库。

典型歧义场景：

- 同一别名对应多个地区；
- 轻度模糊同时命中多个高相似候选；
- 简称过短，无法唯一确定层级或归属。

原则：

- 宁可澄清，不可误查。

## 与当前仓库实现的关系

- `apps/agent/app/repositories/soil_repository.py` 负责从 `region_alias` 读取启用映射。
- `apps/agent/app/services/region_service.py`
  - 先加载数据库中的别名映射；
  - 再补上事实表中的规范全称；
  - 按精确匹配与轻度模糊匹配进行候选决策。
- `RegionResolveService` 的职责是“解析后再校验该地区是否真实存在于事实表中”。

## 验收用例

- `南京最近一个月的数据`：补全为 `南京市`
- `南通最近7天墒情怎么样`：补全为 `南通市`
- `如东最近怎么样`：补全为 `如东县`
- `南晶最近一个月的数据`：唯一高置信命中时允许补全为 `南京市`
- `新区最近怎么样`：如果对应多个候选，必须澄清，不直接查询

## 边界说明

- 这不是通用全文检索系统，也不是开放式地名推荐系统。
- 它只服务于当前墒情业务的受控地区解析。
- 若未来出现跨省、市县重名、乡镇别名爆炸增长等问题，需要再评估是否引入专门地区维表或更强的实体解析方案。
